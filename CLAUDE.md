## Що це за проєкт

Автоматизація для клієнта (СТО/автосервіс): дзвінки менеджерів через Binotel (АТС) автоматично транскрибуються через OpenAI, класифікуються (успішність / найслабший етап / оцінка) і **зберігаються в Postgres із прив'язкою до конкретного менеджера**; окремий Telegram-бот читає ту саму БД і віддає звіти/статистику/архів. Раніше було на n8n; переписано на чистий Node.js, деплой — VPS (AlmaLinux) під pm2: два процеси (cron-інжест + persistent-бот) з одного репо.

**Два компоненти (монорепо):**
1. **Інжест** (`src/jobs/`, pm2 cron-джоб `obv-poller`, `*/15`) — збір, транскрипція, класифікація і атрибуція дзвінків у БД. Звітів тут немає свідомо.
2. **Бот звітності** (`src/bot/`, pm2 persistent-процес `obv-bot`) — інтерактивний Telegram-бот (grammy, long-polling): авто-звіти, статистика менеджера, архів розмов, база знань (RAG). Читає ту саму БД. **Той самий єдиний бот** `@OBVCarServiceWork_bot`, що й алерти інжесту.

Раніше в інжесті були PDF-звіти в Telegram + Google Sheets — видалено. Google Sheets вирізано повністю. **PDF також прибрано:** звіт менеджера тепер **доказовий** (evidence-first) — короткий текст findings + **аудіо-фрагменти** з реальних дзвінків, доставляється в Telegram (не PDF). Деталі — розділ «Доказовий звіт» нижче.

Пріоритети клієнта, які визначили архітектурні рішення: мінімізація регулярних витрат (усе на безкоштовних/копійчаних тарифах, де можливо) і надійність (жодні дані не повинні губитись мовчки).

## Структура репо

Монорепо, feature-based. Стандарт узгоджено під майбутній окремий бот звітності:

```
src/
  core/     — спільне ядро (використовують і jobs/, і bot/):
              store, binotel, transcribe, elevenlabs, classifyCall, analyzeCall (per-call MAP),
              quoteMatch (fuzzy цитата→таймкод), identifyManager, telegram, retry
  jobs/     — задеплоєний cron-джоб (інжест): index.js (entrypoint) + pollNewCalls + processCalls
  scripts/  — одноразові інструменти: backfill, testSingleCall, reattributeShared, retranscribeRecent, backfillAnalysis
  bot/      — Telegram-бот звітності (grammy, окремий токен, окремий сервіс):
              index.js (entrypoint) + keyboards + stats + dynamics (екран росту) + archive + report + segments (кеш аналітики по відрізках) +
              analyze (REDUCE + self-consistency) + audioClip (ffmpeg-нарізка) + prompt + rubric + kb + time + ui
              (див. src/bot/README.md)
```

Коли пишеться нижче просто "processCalls.js", "store.js" тощо — шукати за цією мапою (`core/`, `jobs/` чи `bot/`).

## Архітектура

**Один cron-джоб** (pm2 `obv-poller`, `cron_restart: */15 * * * *`, `JOB_TYPE=poll`) робить усе:

1. **`retryPendingCalls()`** (`processCalls.js`) — спершу ретраїть усе, що застрягло в `pending_calls` з минулих разів
2. **Поллінг** (`pollNewCalls.js`): чекпоінт (`app_state.last_polled_until`) → зараз. Не фіксоване "останні N хв" — щоб пропущений/затриманий cron-запуск не створював діру. `listCallsForPeriod` (Binotel) обмежений 24 год на запит, тому діапазон ріжеться на шматки по 23 год (`processCalls.js: splitIntoChunks`)
3. Для кожного нового дзвінка: якщо `durationSec <= 0` — пропустити (не відповіли); якщо `recordingStatus != 'uploaded'` — в `pending_calls` на ретрай; інакше — `getCallRecordUrl` → `transcribeAudio` (див. «Транскрипція» нижче, повертає `{transcript, segments}`) → **резолвинг імені оператора** (див. нижче) → **per-call MAP** `analyzeCallBehaviors` (`core/analyzeCall.js`, `gpt-4o-mini`: визначає **мету дзвінка** `callPurpose` й, лише для продажних, тегує поведінки менеджера + дослівні цитати, КОД привʼязує таймкоди через `quoteMatch.findQuote`) → **ГЕЙТ ЗА МЕТОЮ:** `classifyCall` (легкий запит: успішність / найслабший етап / оцінка 1-10) виконується **ЛИШЕ якщо `callPurpose === 'sales'`**; для непродажних (`info`/`other`) `classifyCall` **не викликається взагалі** (нуль ресурсу на оцінку ефективності), а `is_success`/`weakest_stage`/`communication_score` = NULL → `saveCall` в Postgres (транскрипт + `segments` + `behaviors` + `call_purpose` + класифікація-або-NULL + `manager_name` разом). **Порядок навмисний** (Задача 1): мета визначається ПЕРШОЮ, і ефективність оцінюється тільки на продажних дзвінках. MAP не фатальний: якщо впав — мета невідома → трактуємо як `sales` (робимо `classifyCall`, щоб не втратити оцінку на збої) і зберігаємо без `behaviors` (беклог полагодить пізніше).

**Транскрипція (`core/transcribe.js` + `core/elevenlabs.js`).** **Первинний шлях — ElevenLabs STT (Scribe):** `POST /v1/speech-to-text` робить транскрипцію + **розділення мовців в одному виклику**. **Стерео vs моно:** `core/audioMeta.js: probeChannels` (ffprobe) визначає к-сть каналів; для **стерео** вмикаємо мультиканальний режим (`use_multi_channel:true`, `diarize:false`, `multichannel_output_style:combined`) — кожен канал = окрема сторона, тож поділ мовців **ідеальний** (беремо прямо з ElevenLabs, без контент-здогадок; слова несуть `speaker_id`/`channel_index`). Для **моно** — контент-діаризація (`diarize:true`), як раніше. Роль «хто менеджер» усе одно вирішує `pickManagerSpeaker`, але на чисто розділеному стерео-тексті це набагато надійніше. Нема ffprobe → тихо працює як моно. З `words[]` (кожне слово має `speaker_id`) збираємо репліки (`buildTurns`) і зберігаємо **готовий діалог «Менеджер:/Клієнт:»** прямо як `transcript`. **Хто саме менеджер (`pickManagerSpeaker(turns, speakerIds, managerName)`)** — шукаємо НАШОГО працівника (не «того, хто грає роль оператора»: на вихідному дзвінку наш менеджер сам телефонує й звучить як замовник — рольова логіка там плутається). Відоме імʼя менеджера прокидається сюди (`processCalls` → `call.employeeName`; скрипт перетранскрибації → `op.name`). Шари, найнадійніший перший: (1) **самопредставлення** `selfIntroManager` — детерміновано: якщо мовець каже «це/это `<Ім'я>`», «`<Ім'я>` вас турбує/беспокоит», «мене звати `<Ім'я>`» → це наш менеджер (ловить кейс, коли менеджер сам телефонує); (2) **LLM** по всьому діалогу, з іменем переформатований на «хто з мовців — наш працівник «`<Ім'я>`»» (або, без імені, на «хто оператор СТО»), + reasoning/confidence; (3) **keyword-евристика** `heuristicManager` (оператор-маркери uk/ru) і перший мовець — як резерв. ⚠️ Content-методи мають стелю (дзвінки без імені + неоднозначні ролі); остаточно надійний шлях — **голосовий відбиток** (планова задача, див. статус). Тобто розділення мовців відбувається на інжесті, а не при відкритті архіву → архів миттєвий. **`transcribeDiarized` повертає `{ transcript, segments }`:** окрім готового рядка, зберігаємо `segments` — репліки з **таймкодами** `start/end` кожного слова (`buildTurns` тепер їх НЕ викидає) у форматі `[{role:'manager'|'client', text, start, end}]` (колонка `calls.segments` JSONB). Таймкоди потрібні доказовому звіту, щоб вирізати аудіо-фрагмент навколо цитати. На OpenAI-fallback (без діаризації) `segments = null` → цитати без аудіо. **Fallback — OpenAI** (якщо нема `ELEVENLABS_API_KEY` / API впав / вичерпано кредити): стара логіка — транскрипція з укр-ухилом → детекція spoken-vs-written (суржик = uk) → перетранскрибація з примусовою мовою, якщо не збігається (фікс «українську записало як російську»). `CALL_LANGUAGE` (env) форсить мову для обох шляхів. Тариф ElevenLabs для ~50 дзв/день — **Scale** (STT = 330 кредитів/хв; ~1,5 млн кредитів/міс).

Ніякої звітності на кожному запуску вже немає — це кінець флоу інжесту.

### Бот звітності (`src/bot/`, окремий persistent-процес під pm2, long-polling grammy)

Персистентний процес (не cron), стартує `bot.start()`.

**Система ролей / контроль доступу (`bot/access.js` + `bot/roles.js`, таблиця `bot_users`).** Доступ більше НЕ керується env-allowlist'ом — ролі живуть у БД. Ролі — фіксований ENUM (без «додати роль»):
- **director** та **marketer** — однаковий ПОВНИЙ доступ (admin-tier): **доказовий звіт** по всіх, архів, база знань (питання+файли), промпт аналізу, керування ролями.
- **manager** — «📊 Моя статистика» (ЛИШЕ числовий блок по собі + свій телефон, **БЕЗ** доказового звіту й AI-аналізу) і база знань (питання). Ідентичність — через `bot_users.operator_name` (директор лінкує при додаванні), тож у статистику по собі природно входять і дзвінки зі спільних номерів, атрибутовані на те саме ім'я. **Доказовий звіт з аудіо — інструмент керівника, менеджеру недоступний.**
- **mechanic** — лише база знань (питання).

`access.js`: `getUser` (роль по `telegram_id`, кеш із `invalidateRole`), `canAccess(role, feature)` + `featureOf(ctx)` — центральний gate в auth-middleware (класифікує кожен апдейт у фічу й пускає/блокує). `mainMenu(role)` і нативні команди (`setMyCommands` зі `scope:{type:'chat'}`) підбираються під роль. **Bootstrap:** `seedDirectors()` на старті сідить user-id з `TELEGRAM_BOOTSTRAP_CHAT_IDS` як директорів (щоб не залочитись), відсіюючи негативні (групові) id — ролі прив'язані до `ctx.from.id` (user).

**Керування ролями (`roles.js`, admin-only, блок «👥 Ролі» / `/roles`):** кнопки Маркетологи/Менеджери/Механіки (директорів UI навмисно не чіпає — захист seed-власника від видалення). Роль → список людей + «➕ Додати» → людина → «🗑 Видалити». **Додавання** (Telegram не дає ботам шукати за `@username` — це навмисне обмеження приватності): (1) `request_users` — обрати колегу з контактів, отримуємо `user_id`(+ім'я/username) МИТТЄВО, без потреби, щоб людина спершу писала боту (Bot API 7.2, grammy `Keyboard.requestUsers`); (2) поділитися контактом (є `user_id` → active, інакше pending за телефоном); (3) ввести телефон → pending. **Pending → active:** незнайомець ділиться СВОЇМ контактом на `/start`, `activatePendingByPhone` матчить за останніми 9 цифрами. Для manager після додавання — вибір оператора з roster (`getOperators`) → `operator_name`. ⚠️ Навіть маючи `user_id`, бот не може написати людині першим, поки вона не натисне Start (правило Telegram) — тож особисті авто-звіти неможливі; «Мій звіт» менеджер тягне сам інтерактивно.

**Навігація:** (1) нативні команди — кнопка «Menu» біля поля вводу, **список підбирається під роль** (`setMyCommands` зі `scope:{type:'chat'}`): admin — `/menu /stats /archive /ask /files /report /prompt /rubric /roles /settings`; manager — `/menu /myreport /ask`; mechanic — `/menu /ask`; (2) inline-меню в повідомленнях (`keyboards.js: mainMenu(role)`) + inline «« Меню» на всіх екранах. Постійну reply-клавіатуру (`quickKeyboard`/`QUICK`) **прибрано** (забагато UI-навантаження) — `/start` шле `remove_keyboard`, щоб очистити її в старих клієнтів. Нативний список команд свідомо ширший за inline-меню: **`/files` (Файли) і `/settings` (Налаштування) є ЛИШЕ в нативному списку**, inline-кнопок для них немає (свідомо, щоб не перевантажувати меню). Inline-кнопки admin-меню: «📊 Статистика менеджера», «🗂 Архів розмов», «📚 База знань» (`kb:ask`), «🔄 Звіт зараз», «👥 Ролі».

**Модель «активного екрана»** (`bot/ui.js: showScreen()`, замінила прямі `editMessageText`): меню/пікер завжди живе в НАЙнижчому повідомленні, щоб фокус юзера (низ чату) збігався з дією. Якщо натиснуте меню ще внизу — редагуємо на місці (плавно, без спаму); якщо його «поховало» під контентом (транскрипт / відповідь KB / текст промпту) — надсилаємо свіже меню внизу й видаляємо старе. `installMessageTracker` (API-transformer) веде id останнього надісланого ботом повідомлення на чат (щоб знати, чи меню ще внизу); `ctx.session.screenId` — id активного екрана (для видалення старого). Контентні дії (перегляд промпту тощо) після себе ре-планують меню вниз. Best practice підтверджено пошуком (стандарт — edit-in-place; тут адаптовано під контент-важкий бот).

**Доказовий звіт (evidence-first, map→reduce з кешем).** Замість узагальненої прози — findings, кожен підкріплений **≥3 реальними, релевантними цитатами**, до негативних — **аудіо-фрагменти**. Шари:
- **MAP** (`core/analyzeCall.js`, per-call, `gpt-4o-mini`, на інжесті) — спершу визначає **тип дзвінка** (`callPurpose`: `sales`/`info`/`other`) і зберігає його в `calls.call_purpose`. **Поведінки тегуються ЛИШЕ для `sales`-дзвінків** (інформаційний «машина готова» / підтвердження запису / клієнт уточнює статус → 0 поведінок, щоб непродажна розмова не стала «помилкою продажу»). Цитати приймаються ЛИШЕ якщо знайдені в сегменті **менеджера** (`quoteMatch.findQuote({requireRole:'manager'})` → клієнтську репліку не можна видати за менеджерську). КОД привʼязує таймкоди. Кеш у `calls.behaviors` (`analysis_version=2`). **Раз на дзвінок** → звіти за період не переаналізовують дзвінки (закрило «брудну» генерацію).
- **REDUCE** (`bot/analyze.js: reduceFindings`, per-report, `OPENAI_REPORT_MODEL`=gpt-4o) — тягне кешовані `behaviors` **лише продажних** дзвінків періоду (`buildCandidates` пропускає `info`/`other`) + метрики, кластеризує в findings. Модель посилається на докази **лише за id** → вигадати цитату не може.
- **Код-верифікація** (`assembleFindings`, чиста, юніт-тестована): цитата re-верифікується проти сегментів **менеджера** свого дзвінка; один доказ підпирає лише ОДИН finding (дедуп); finding з <3 доказами викидається; «спершу помилки».
- **Перевірка релевантності** (`verifyFindingsRelevance`, окремий LLM-прохід `OPENAI_REPORT_MODEL`) — суворий рецензент вирішує, які цитати **справді доводять** claim; нерелевантні відкидаються, finding з <3 підтвердженими валиться. Це закриває кейс «цитата існує, але не про те». Фейл API → graceful fallback на assembled (звіт не гине).
- **Аудіо** (`bot/audioClip.js`, системний ffmpeg) — лише під негативні findings, ≤3 кліпи/finding, ±`AUDIO_CLIP_PAD_SEC` навколо таймкоду; запис Binotel тягнеться на льоту й кешується на дзвінок у межах звіту. Нема ffmpeg → звіт іде текстом (preflight, не падає).
- **Звіт рендериться ЗАВЖДИ** (`deliverReport`): числовий header з розбивкою **всього / продажних / інформаційних** (конверсія й найслабший етап — по продажних) + findings; якщо findings нема — чіткий рядок «критичних патернів не зафіксовано» / «продажних дзвінків не було», а не «недостатньо матеріалу».
- **Доступ:** доказовий звіт бачать ЛИШЕ director/marketer.

1. **Звіти** (`report.js` + `segments.js`) — планувальник (`setInterval` 30с звіряє київський час зі слотами). **Слоти — з `/settings` → «🕒 Час звітів» (`app_state.report_times`), не env** (дефолт `13:00,19:30`; порожній = нема). **Не окремий cron, а in-process інтервал** persistent-бота. **ПЕРСИСТЕНТНА АНАЛІТИКА ПО ВІДРІЗКАХ (Задача 3, Фаза 1):** дорогий REDUCE кешується в таблиці `report_segments` по **(менеджер × день-обмежений відрізок)**, тож звіти НЕ переаналізують уже пораховане. Відрізки — межі доби Київ, розбиті слотами (`time.js: kyivDaySegments` → `[00:00,слот1),[слот1,слот2),…,[остСлот,24:00)`, без дір). `assembleReport(name,start,end)` (`segments.js`) = **заморожені `scheduled`-відрізки (reuse або порахувати+зберегти) + живий "хвіст"** `[остання межа, now]`; числа завжди live по періоду. **Self-consistency:** заморожений відрізок аналізується `SEGMENT_CONSISTENCY_PASSES`(деф.3) разів, лишаються findings, підтверджені більшістю прогонів (`analyze.js: reduceFindingsConsistent`/`corroborate`) — він рахується РАЗ і замерзає. **Grace:** слот віддається лише через `SEGMENT_GRACE_MIN`(деф.10) хв після межі (щоб ретраї `pending_calls` встигли). **Self-heal:** якщо в НЕДАВНІЙ (≤24 год) заморожений відрізок доїхав спізнілий дзвінок (`call_ids` змінились) — його тихо перераховують. Планувальник (`maybeSendScheduledReport`) віддає кожен ще недоставлений слот сьогодні (дедуп `app_state.delivered_report_slots`), сегмент = `[попередня межа, слот]`. **«Звіт зараз»/`/report`** (`sendManualReport`, за сьогодні, стан розкладу не чіпає) → `assembleReport` reuse + **дедуп подвійного кліку** (ефемерний `manual_tail` з тим самим `call_ids` віддається без повторного аналізу). Звіт рендериться **блоками** (кожен відрізок — свій блок findings, підписаний інтервалом → основа для відстеження РОСТУ). **Мульти-денні періоди (Фаза 2):** тиждень/місяць/квартал у «Статистиці менеджера» = `buildTrendReport` (`mode:'trend'`) — числовий **тренд по днях** (`getDailyTrend`, live SQL) + findings з **уже заморожених** scheduled-відрізків (`getScheduledSegmentsInRange`, **REUSE-ONLY**, останні ≤3 з findings; історичні відрізки НЕ рахуються на льоту — інакше місяць = 60-90 сегментів × self-consistency = вибух вартості). `deliverManagerReport({mode})`: `daily` (сегментна збірка+хвіст) / `trend` (мульти-день) / `live` (легасі single-reduce fallback).
2. **Статистика менеджера** (`stats.js`, admin) — клік на менеджера → **екран «Динаміка/Ріст»** (головний, `stat:op` → `showDynamics`, дефолт тижні; toggle тижні⇄місяці `stat:dyn:<bucket>`). Це НЕ разовий аналіз ефективності, а **відстеження РОСТУ**: `bot/dynamics.js: buildDynamicsText` рендерить моноширинну траєкторію по бакетах (`store.getBucketedTrend` — live SQL по `calls`, ретроактивно на всю історію; date_trunc week/month у Києві, MODE слабкого етапу) — конверсія/бал/обсяг з дельтами ↑↓, **еволюція слабкого етапу** по бакетах + вердикт (РІСТ✅/СПАД⚠️/змішана/без змін) і найчастіша слабина. Текст-онлі, без LLM → миттєво. **Дрілдаун «📊 Звіт за період →»** (`stat:rep` → пікер періоду → `stat:go`) веде до класичного доказового звіту: `deliverManagerReport` (audio) — числовий header + findings + аудіо-кліпи + фрази (day → сегментний кеш; тиждень/місяць/квартал → тренд Фази 2). Кнопки нотаток (`manager_notes`). Слабкі етапи — спільна таксономія `core/stages.js: SALES_STAGES`. **«Моя статистика» менеджера (`me:go`) — ЛИШЕ числовий блок** по собі, БЕЗ динаміки/findings/аудіо.
3. **Архів розмов** (`archive.js`) — менеджер → період → список (пагінація) → дзвінок: **розмова у форматі діалогу** + "🎧 Прослухати" (свіжий `getCallRecordUrl` з Binotel → `sendAudio`). Для нових дзвінків діалог «Менеджер:/Клієнт:» уже готовий (ElevenLabs-діаризація на інжесті) — показується **миттєво, без AI-запиту**. Для старих/fallback-транскриптів (суцільний текст, без міток; `looksDiarized()` це визначає) залишився запасний on-demand `bot/dialogue.js: formatDialogue` (OpenAI, ~10-20с, фейл → сирий текст). У кнопці **продажного** дзвінка успішність позначена 👍/👎 + бал; **непродажні** (`info`/`other`) — нейтрально `ℹ️ інформаційний`/`службовий` (без 👍/👎 і бала, бо оцінки ефективності в них немає — див. Задачу 1). У списку менеджерів біля імені — період активності `dd.mm.yy-dd.mm.yy` (`operatorListKeyboard(..., {showDates:true})`). У кнопці дзвінка успішність позначена 👍/👎 (не ✅/❌). У списку менеджерів біля імені показується період активності `dd.mm.yy-dd.mm.yy` (`operatorListKeyboard(..., {showDates:true})`): від найранішого обробленого дзвінка (`getOperators` → `MIN(start_time)`, проксі «коли Binotel уперше побачив ім'я» — реальної дати реєстрації Binotel API не віддає) до сьогодні. Тільки в архіві; у статистиці той самий пікер без дат.

**Аліаси імен операторів** (`bot/operators.js: displayName`) — мапа «сирий `manager_name` → людське ім'я», застосовується ЛИШЕ на рендері (кнопки, заголовки, деталі дзвінка, звіт). БД і callback-data лишаються сирими → групування/запити/атрибуція не змінюються, мапа повністю оборотна. Немає вбудованого дефолту (колишній `0674738200` → «Богдан» прибрано разом із виключенням цього номера з інжесту — див. «Ідентифікація оператора»). Розширюється env `OPERATOR_ALIASES` (пари `номер=Ім'я` через кому). **Емоджі в пікері операторів** (`keyboards.js: operatorLabel`): іменований/аліасований оператор — `📱` (клієнт просив показувати як «персональний номер»); лише неатрибутований спільний номер лишається `☎️ Спільний`.

**Список операторів (кнопки), єдиний компонент** (`keyboards.js: operatorLabels` + `operatorListKeyboard`) — і «Статистика менеджера», і «Архів розмов», і пікер оператора при додаванні ролі менеджера (`roles.js: operatorPickScreen`) будують підписи кнопок через ОДНУ функцію `operatorLabels()`. Вона ж робить **best-effort наближення лівого вирівнювання**: Telegram Bot API не має властивості text-align для inline-кнопок (клієнт завжди центрує підпис, це обмеження платформи, не боту) — тому кожен підпис у списку доповнюється праворуч символом BRAILLE PATTERN BLANK (`⠀`, не пробіл — не обрізається як trailing whitespace) до довжини найдовшого підпису в ЦЬОМУ СПИСКУ; клієнт центрує «текст+доповнення» як єдиний блок, і видимий текст візуально зміщується ліворуч від центру. Не піксель-перфектно (пропорційний шрифт, відносно найдовшого лейбла в списку, а не справжнього краю екрана) — це наближення, не гарантія.
4. **База знань** (`kb.js`, кнопка «📚 База знань» → `kb:ask`, команда `/ask`) — **RAG** по завантажених посібниках. Власник надсилає боту документ (PDF `unpdf` / DOCX `mammoth` / TXT) → **бот питає, ДЛЯ КОГО файл** (категорія-audience: 🔧 Механікам / 💼 Менеджерам / 👥 Обом; файл стешиться в `ctx.session.pendingKbDoc`, інжест відкладений до вибору `kb:aud:*`) → **посторінковий витяг** (`extractPages`: PDF `mergePages:false` → текст на кожну сторінку; DOCX/TXT — сторінок нема, `page=null`) → **page-aware нарізка** (`chunkDocument`, ~2400 симв., трекає діапазон сторінок на чанк) → embeddings (`text-embedding-3-small`) → Postgres (`kb_docs.audience` + `kb_chunks` з `vector(1536)` + `page_start/page_end`, **pgvector**).
   - **Пошук (покращений, multi-query):** `retrieve()` — `expandQueries` просить модель дати 3 переформулювання питання, embed'ить оригінал+варіанти, шукає кожен (`searchKbChunks`, по 8), мерджить за `chunkId` (найкраща відстань), бере топ-8. Це полагодило слабку повторюваність («З чого складається двигун» vs «Як працює двигун»). Розширення — best-effort (падіння → лише сире питання).
   - **Категорії/доступ (audience):** `searchKbChunks(emb, k, audiencesForRole(role))`: `manager` → `manager`+`both`, `mechanic` → `mechanic`+`both`, `director`/`marketer` (admin) → усе (фільтр `null`). Механік не отримає посібник продажів і навпаки.
   - **Відповідь (структурована, `ANSWER_SCHEMA`):** модель повертає `{answer, usedSources[], usedGeneralKnowledge}`. **Дозволено ~10% загальних знань:** якщо фрагменти покривають лише частково — модель доповнює достовірними загальними знаннями (не галюцинуючи, не суперечачи посібникам) і ставить `usedGeneralKnowledge=true`. Відповідь — простим текстом (модель НЕ додає джерела сама).
   - **Блок джерел (детермінований, ОБОВ'ЯЗКОВИЙ):** будується в КОДІ з `usedSources` (`buildSourcesFooter`), а не моделлю. Групується по файлу, сторінки мерджаться (`pagesLabel` → `стор. 3–4, 12`). Назва файлу — **гіперпосилання-діплінк** `https://t.me/<bot>?start=kbdoc_<docId>` (клік → `/start kbdoc_<id>` у `index.js` → `openKbDocById` пересилає оригінал по `file_id`, з перевіркою audience). Відповідь шлеться **HTML** (`sendLong(..., {parseMode:'HTML'})`) — назви з `_` не ламають розмітку. Якщо додано загальні знання — окремий рядок-примітка.
   - Екран «📚 Файли»: у списку біля кожного файлу — категорія; деталі → 📄 Відкрити оригінал / **🔁 Змінити для кого** (`kb:audset`→`kb:audput`) / 🗑 Видалити (каскад) + «➕ Завантажити новий». `migrateKb()` guard'иться: нема pgvector → KB вимкнено, решта бота живе. **KB-екрани меню — простий текст без Markdown** (назви файлів з `_`); а от сама ВІДПОВІДЬ на питання — HTML (для діплінків). Після відповіді бот **лишається в режимі питань** (`awaiting=kb_question`); вихід — `/menu`. ⚠️ Сторінки показуються лише для файлів, залитих ПІСЛЯ цього оновлення (старі чанки мають `page_start=NULL`).
5. **Промпт аналізу** (`prompt.js`, команда `/prompt`) — власник переглядає / змінює / скидає **guidance-промпт reduce** (тон/формулювання claim/why/action + готові фрази). Меню: 👁 переглянути, ✏️ змінити (`awaiting=prompt`), ↩️ скинути. Зберігається в `app_state.analyze_prompt`; `analyze.js: getAnalyzePrompt()` повертає його або `DEFAULT_REPORT_GUIDANCE`. ⚠️ **Промпт керує лише формулюваннями — СТРУКТУРУ й правило доказів (JSON-схема findings, ≥3 підтверджених цитат, дедуп, аудіо для негативу) забезпечує КОД (`assembleFindings`), не промпт.** Тобто хоч би що написав власник, вигадані/недоведені твердження не проходять. Старий прозовий `DEFAULT_ANALYZE_PROMPT` прибрано; легасі-значення `app_state.analyze_prompt` одноразово скидається в `migrate()` (guard `analyze_prompt_v2`), щоб діяв новий дефолт.

   **Рубрика оцінки комунікації** (`rubric.js`, команда `/rubric`, admin-only — пара до `/prompt`) — власник переглядає / змінює / скидає **рубрику `communicationScore`** (критерії, за якими AI виставляє бал 1-10 КОЖНОМУ дзвінку на інжесті — не в звіті). Меню: 👁 переглянути, ✏️ змінити (`awaiting=rubric`), ↩️ скинути. Зберігається в `app_state.score_rubric`; `core/classifyCall.js: getScoreRubric()` повертає його або `DEFAULT_SCORE_RUBRIC` (анкерована шкала 9-10/7-8/5-6/3-4/1-2). ⚠️ Редагується ЛИШЕ формулювання критеріїв — шкалу 1-10, JSON-структуру й **таксономію етапів (`core/stages.js`, 4 етапи, НЕ редагована)** забезпечує КОД. На відміну від `/prompt` (guidance reduce у звіті), рубрика впливає на per-call бал ще на етапі обробки.

6. **Налаштування** (`settings.js`, команда `/settings` — **лише нативна, БЕЗ inline-кнопки**, admin-only) — керування двома списками отримувачів (обидва в `app_state` як JSON `{id,name}[]`, тому повідомлення розсилається кільком людям одразу): **«⚠️ Сповіщення про поломки»** (`alert_recipients`, куди `core/telegram.js: sendAlert` шле алерти інжесту) і **«📊 Щоденні звіти»** (`report_recipients`, куди йдуть авто-звіти). На екрані списку: кнопка `🗑 <ім'я>` на кожного (видалення одним тапом), «➕ Додати» → reply-клавіатура (`request_users` «Обрати з контактів» / «Поділитися контактом» / надіслати числовий ID чату — останнє для груп на кшталт `-5426146652`). Отримувач = Telegram chat_id (юзер, що натиснув Start, або група). Store-обгортки — `getRecipients/addRecipient/removeRecipient(kind)`. Крім двох списків отримувачів, тут же **«🕒 Час звітів»** (`app_state.report_times`, `getReportTimes/addReportTime/removeReportTime`): київські слоти щоденних звітів, кероване кнопковим пікером **година → хвилина (крок 10 хв)** — формат завжди канонічний `HH:MM`, ввести невалідне неможливо. Слотів може бути скільки завгодно; порожньо = звітів нема; дефолт (поки не задано) — `13:00,19:30`. Callback'и: `set:times` (список), `set:timeadd`→`set:timeh:<h>`→`set:timem:<h>:<m>` (додати), `set:timedel:<HHMM>` (прибрати). Gate — `featureOf` повертає `settings` для `set*` / `/settings` (admin-only). ⚠️ Замінило env-змінні `TELEGRAM_CHAT_ID`, `BOT_REPORT_CHAT_ID` і `BOT_REPORT_TIMES` (усі прибрані). Хендлери `message:contact`/`message:users_shared` у `roles.js` і `settings.js` пропускають чужий апдейт далі через `next()` — ланцюжок: roles → settings → save_phone (в `index.js`).

Бот НЕ пише в `calls`/`pending_calls` — лише читає; пише в `manager_notes`, `kb_docs`/`kb_chunks`, `bot_users` (ролі) і ключі `app_state` (звітні + `analyze_prompt` + `alert_recipients`/`report_recipients`).

### Ідентифікація оператора (`resolveManagerName` в `processCalls.js`)

**Джерело правди про ІСНУВАННЯ операторів — Binotel** (таблиці `managers` немає, клієнт керує операторами в Binotel). Але з 2026-07-24 **джерело правди про ТЕ, ЧИЙ це дзвінок на персональному номері — сам номер (`internalNumber`), а не ім'я, яке в цей момент повертає Binotel.**

- **Особистий номер** (903/904/905) → `manager_name` = **канонічне ім'я з фіксованої мапи `PERSONAL_OPERATORS`** (`processCalls.js`, дефолт `903=Роман,904=Андрій,905=Володимир`, override через однойменний env). Ім'я з Binotel (`employeeData.name`) для цих номерів **ігнорується повністю** — ні для збереження, ні для `identifyManager`.
- **Спільна слухавка** (номер у `SHARED_EXTENSIONS`, дефолт `901,902`) або будь-який ІНШИЙ номер без імені (не в `PERSONAL_OPERATORS`) → Binotel не знає, хто відповів, тому оператор визначається **з транскрипту**: `identifyManager.js` окремим OpenAI-запитом обирає, ХТО зі списку відомих операторів представився. Список кандидатів (**roster**) — динамічний, `getOperatorRoster()` = distinct не-числові `manager_name` з `calls`. AI обмежений `enum`'ом кандидатів, тож повертає точно одне з тих імен (сам зводить розмовні/RU-варіанти) або `null`. Збіг → це ім'я. Нема збігу/імені → лишається голий номер як мітка.
- **Виключені номери** (`EXCLUDED_EXTENSIONS`, дефолт `0674738200` — мобільний директора) → дзвінок **пропускається на самому вході** (`processOneCall`/`retryPendingCalls`), ще до транскрипції/аналізу: не зберігається в `calls`, не лишається в `pending_calls`. Директор — не продажник, і оцінка ефективності по ньому не потрібна.

⚠️ **Баг 2026-07 (виправлено):** для персональних номерів 903/904 Binotel періодично повертав `employeeData.name` ПУСТИМ (з невідомої причини — можливо, тимчасовий десинк після редагування імені співробітника в адмінці Binotel). Старий код тоді трактував це як спільну лінію і жену дзвінок через `identifyManager` — а той **іноді вгадував НЕПРАВИЛЬНОГО менеджера** за голосом/контентом (дзвінки з номера Романа/Андрія приліплювались до Володимира). Окремо, номер директора (0674738200) ніколи не мав `employeeData` і завжди йшов через `identifyManager`, регулярно забруднюючи статистику реальних менеджерів. Фікс: персональні номери відв'язані від `employeeName`/`identifyManager` повністю (мапа за номером), директорський номер виключений з інжесту. Історичні дані виправлено скриптом `normalize:operators` (див. «Команди»). **Урок:** для персональних номерів ідентифікація мусить бути за стабільним `internalNumber`, а не за іменем, яке може зникнути/змінитись на боці Binotel.

Тобто roster сам «наповнюється» з Binotel (для 901/902): як тільки менеджер зробив дзвінок з особистого номера, його канонічне ім'я стає кандидатом для матчингу спільних дзвінків. **Клієнт попросить операторів представлятись** на спільних — від цього залежить якість атрибуції 901/902.

`reattributeShared.js` (`npm run reattribute:shared`) — переганяє вже збережені спільні дзвінки (з голим номером) через ту саму AI-ідентифікацію по збережених транскриптах. `normalizeOperators.js` (`npm run normalize:operators`) — одноразовий фікс історії після бага вище (не плутати: це не про 901/902, а про персональні номери + видалення директора).

### Надійність (свідомі архітектурні рішення, не міняти без причини)

- **Чекпоінт замість вікна** — гарантує відсутність дір при збоях cron
- **`pending_calls`** — дзвінок, що впав на будь-якому кроці, ретраїться на кожному наступному `poll` (не втрачається), до `MAX_PENDING_ATTEMPTS` (20), потім `status='failed'` + Telegram-алерт, рядок лишається в таблиці для ручної перевірки (нічого не видаляється мовчки)
- **`withRetry`** (`retry.js`) — обгортає кожен зовнішній виклик (Binotel/OpenAI/Telegram), 2-3 спроби з наростаючою паузою
- **Telegram (інжест) — тільки алерти.** `core/telegram.js` лише `sendMessage`/`sendAlert` (вихідні), для сповіщень про збої й невідомих менеджерів. `sendMessage` ділить текст на шматки ≤4096 і має markdown→plain fallback. Шле через той самий єдиний бот `@OBVCarServiceWork_bot` (вихідні виклики не конфліктують з `getUpdates` бота звітності).

### Схема БД (локальний Postgres 18 на VPS, `core/store.js: migrate()`)

**SSL:** `store.js` вмикає SSL умовно (`sslConfig()`): для `localhost`/`127.0.0.1` або `sslmode=disable` → без SSL (локальний PG18 SSL не має); для віддаленого/managed (Neon, `sslmode=require`) → як раніше. Це полагодило «server does not support SSL connections» після переїзду з Neon на локальний PG.

- **`calls`** — `general_call_id` (унікальний), `internal_number`, `manager_name` (ім'я оператора — ключ групування скрізь), `start_time`, `duration_sec`, `transcript`, `is_success`, `weakest_stage`, `communication_score`, **`segments`** JSONB (репліки з таймкодами `[{role,text,start,end}]` для нарізки аудіо; NULL на OpenAI-fallback/старих рядках), **`behaviors`** JSONB (кешований per-call MAP: `{version, callPurpose, items:[{type,stage,label,quote,start,end,segIndex}]}`; items непорожні лише для продажних), **`analysis_version`** INTEGER (=2; для re-map/беклогу), **`call_purpose`** TEXT (`sales`/`info`/`other`; лише `sales` дає findings, решта — тільки в числову розбивку; NULL = ще не аналізовано, трактується як sales до беклогу — прибрати `npm run backfill:purpose`). **Для непродажних дзвінків `is_success`/`weakest_stage`/`communication_score` = NULL** (з Задачі 1 `classifyCall` не викликається для `info`/`other`); архів показує їх нейтрально (`ℹ️ інформаційний`, без 👍/👎/бала). (Таблиці `managers`, колонок `manager_id` і старого `call_type` (напрямок Binotel) більше немає — `migrate()` ідемпотентно прибирає ці legacy-артефакти; нове `call_purpose` — інша семантика, окрема колонка.)
- **`pending_calls`** — підмножина полів calls + `attempts`, `status` (`pending`/`failed`), `last_error`
- **`manager_notes`** — `id`, `operator_name` (ім'я оператора = `calls.manager_name`), `author`, `note`, `created_at`. Нотатки, які лишає власник у боті (фіча 2)
- **`bot_users`** — контроль доступу (ролі): `id`, `telegram_id` (UNIQUE, NULL поки invite pending), `role` (director|marketer|manager|mechanic), `phone`, `username`, `display_name`, `operator_name` (лінк менеджера до `calls.manager_name` для стати по собі), `status` (active|pending), `added_by`, `created_at`. **Це НЕ воскресіння старої `managers`** (та була про атрибуцію дзвінків) — тут суто авторизація «хто і що може в боті»; Binotel лишається джерелом правди щодо того, хто говорив. Функції в `store.js`: `getBotUser`/`getBotUsersByRole`/`upsertBotUserByTelegram`/`addPendingBotUser`/`activatePendingByPhone`/`setBotUserOperator`/`setBotUserPhone`/`deleteBotUser`/`seedDirector`
- **`kb_docs`** / **`kb_chunks`** — база знань (фіча 4): `kb_docs` (файл + `file_id`/`mime` для повторної відправки оригіналу + `chunk_count` + **`audience`** TEXT `mechanic`|`manager`|`both` — для кого посібник; `NOT NULL DEFAULT 'mechanic'`, що й бекфілить наявні рядки на механіка), `kb_chunks` (`doc_id` FK ON DELETE CASCADE, `ord`, `content`, `embedding vector(1536)`, hnsw-індекс cosine, **`page_start`/`page_end`** — діапазон сторінок чанка для цитування джерел; NULL для DOCX/TXT і для чанків, залитих до цього оновлення). Створюються окремо в `migrateKb()` (потребує `CREATE EXTENSION vector`), не в `migrate()`. Пошук фільтрує за `audience` залежно від ролі того, хто питає (`searchKbChunks(..., audiences)`)
- **`report_segments`** (Задача 3, Фаза 1) — кеш результатів аналітики по `(manager_name × [period_start, period_end] × kind)`: `findings` JSONB, `phrases` JSONB, `stats` JSONB (знімок), `call_ids` JSONB (склад — дедуп/self-heal), `candidate_count`, `analysis_version` (важіль інвалідації), `meta` JSONB (`{rubricHash,promptHash,model,passes}`), `created_at`/`updated_at`. `kind`: `scheduled` (день-обмежений канонічний відрізок — незмінний часовий ряд РОСТУ, аналіз із self-consistency) або `manual_tail` (ефемерний хвіст «Звіт зараз», дедуп по `call_ids`, GC >2 діб). UNIQUE(`manager_name`,`period_start`,`period_end`,`kind`). Пише **бот**, не інжест. Функції — `store.js: getStoredSegment/getLatestManualTail/getScheduledSegmentsInRange/upsertReportSegment/getCallIdsForOperator/deleteOldManualTails`.
- **`app_state`** — key-value: `last_polled_until` (чекпоінт інжесту); `last_report_slot`/`last_report_until` (розклад звітів, пише **бот**); `analyze_prompt` (кастомний guidance-промпт reduce, якщо власник змінив через `/prompt`; нема ключа → `DEFAULT_REPORT_GUIDANCE` з `analyze.js`); `analyze_prompt_v2` (маркер одноразового скидання легасі-прозового промпту в `migrate()`); `score_rubric` (кастомна рубрика `communicationScore`, якщо змінена через `/rubric`; нема ключа → `DEFAULT_SCORE_RUBRIC` з `core/classifyCall.js`); `alert_recipients`/`report_recipients` (JSON `{id,name}[]`); `report_times` (JSON `["HH:MM"]` — київські слоти; керується в `/settings`, фіча 6); `delivered_report_slots` (JSON `["YYYY-MM-DD-HH:MM"]` — дедуп доставлених слотів авто-звіту, замінив однозначний `last_report_slot`); `elevenlabs_balance_state` (`ok`/`low`/`no_permission` — дедуп алерту низького балансу ElevenLabs). Обгортки — `store.js: getStoredAnalyzePrompt/…`, `getStoredScoreRubric/setStoredScoreRubric/clearStoredScoreRubric`, `getRecipients/addRecipient/removeRecipient(kind)`, `getReportTimes/addReportTime/removeReportTime`, `getDeliveredSlots/markSlotDelivered`, `getElevenLabsBalanceState/setElevenLabsBalanceState`

Час: інжест рахує все в UTC (чекпоінт — абсолютний момент), йому Kyiv не треба. **Боту** Kyiv потрібен (слоти звітів + межі періодів) — `src/bot/time.js` рахує через `Intl.DateTimeFormat`, без хардкод-офсету (ламається на EET/EEST). У `core/` спільного `time.js` немає — це суто ботова логіка.

## Env-змінні

Повний список і коментарі — `.env.example`. Ключове:
- `BINOTEL_API_KEY/SECRET`, `BINOTEL_BASE_URL` — ендпоінти підтверджені емпірично на реальному акаунті 2026-07-06 (`stats/list-of-calls-for-period.json` з `startTime`/`stopTime` в unix-секундах, `stats/call-record.json`)
- `OPENAI_ANALYZE_MODEL` — `classifyCall` (бал за редагованою рубрикою `app_state.score_rubric` / `/rubric` + 4 спільні етапи `core/stages.js`), `identifyManager`, **per-call MAP (`analyzeCall`)**, відповіді бази знань, лейблінг мовців (дефолт `gpt-4o-mini`). `OPENAI_REPORT_MODEL` — **reduce доказового звіту** (агрегація в findings; рідко → сильніша модель, дефолт `gpt-4o`). `OPENAI_EMBED_MODEL` — embeddings бази знань (дефолт `text-embedding-3-small`). `OPENAI_TRANSCRIBE_MODEL` — лише fallback-транскрипція (дефолт `gpt-4o-mini-transcribe`)
- `FFMPEG_PATH` (дефолт `ffmpeg`), `AUDIO_CLIP_PAD_SEC` (дефолт `3`) — нарізка аудіо-кліпів доказового звіту (`bot/audioClip.js`). Нема ffmpeg → звіт текстом. `BACKFILL_LIMIT` (дефолт `30`) — глибина `npm run backfill:analysis`
- `SEGMENT_CONSISTENCY_PASSES` (дефолт `3`) — скільки разів проганяти REDUCE на заморожений відрізок; лишаються findings, підтверджені строгою більшістю прогонів (`bot/segments.js`, Задача 3). `SEGMENT_GRACE_MIN` (дефолт `10`) — на скільки хв відкласти віддачу слота авто-звіту, щоб ретраї `pending_calls` встигли потрапити у відрізок до заморозки
- `ELEVENLABS_API_KEY` — **первинний транскрайбер** (STT + діаризація/мультиканал, `core/elevenlabs.js`). Порожньо → fallback на OpenAI-транскрипцію (без діаризації/таймкодів). `ELEVENLABS_STT_MODEL` (дефолт `scribe_v1`), `ELEVENLABS_NUM_SPEAKERS` (дефолт `2`, для моно-діаризації). Заголовок авторизації — `xi-api-key`. `CALL_LANGUAGE` форсить `language_code`. **Стерео** авто-детектиться (`FFPROBE_PATH`, дефолт `ffprobe`) → мультиканальний STT.

  поллінгу, дедуп через `app_state.elevenlabs_balance_state`, шле `sendAlert` тим самим отримувачам, що й помилки): `ELEVENLABS_MIN_BALANCE_USD` (дефолт `2`), `ELEVENLABS_USD_PER_1000_CREDITS` (дефолт `0.22` — API віддає кредити, не долари; калібрується під дашборд). ⚠️ **Ключ мусить мати дозвіл `user_read`** (інакше `/v1/user/subscription` → 401, і бот шле одноразове попередження «додайте user_read»). Алерт спрацьовує ЛИШЕ при зміні стану (перетин порогу вниз), і переозброюється, коли баланс піднявся.
