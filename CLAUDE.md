## ⏳ Активне нагадування (тимчасове — прибрати, коли зроблено)

**Розгорнутий AI-аналіз у «Статистика менеджера» / «Мій звіт» зараз генерується щоразу «брудно»** — повний виклик OpenAI (~20-40с, платно) на кожне відкриття періоду, без кешу. Це свідомий тимчасовий стан: **окрема майбутня задача — оптимізація, кешування й швидкодія** (напр. кеш аналізу по (менеджер+період) з коротким TTL). **Поки цього не зроблено — нагадуй про це в КОЖНІЙ відповіді користувачу.** Коли реалізуємо — прибрати цей блок. (Форматування діалогу в архіві вже НЕ в цьому списку — перейшло на інжест через ElevenLabs-діаризацію, зберігається готовим; on-demand лишився лише як fallback для старих/plain транскриптів.)

## Що це за проєкт

Автоматизація для клієнта (СТО/автосервіс): дзвінки менеджерів через Binotel (АТС) автоматично транскрибуються через OpenAI, класифікуються (успішність / найслабший етап / оцінка) і **зберігаються в Postgres із прив'язкою до конкретного менеджера**; окремий Telegram-бот читає ту саму БД і віддає звіти/статистику/архів. Раніше було на n8n; переписано на чистий Node.js, деплой — VPS (AlmaLinux) під pm2: два процеси (cron-інжест + persistent-бот) з одного репо.

**Два компоненти (монорепо):**
1. **Інжест** (`src/jobs/`, pm2 cron-джоб `obv-poller`, `*/15`) — збір, транскрипція, класифікація і атрибуція дзвінків у БД. Звітів тут немає свідомо.
2. **Бот звітності** (`src/bot/`, pm2 persistent-процес `obv-bot`) — інтерактивний Telegram-бот (grammy, long-polling): авто-звіти, статистика менеджера, архів розмов, база знань (RAG). Читає ту саму БД. **Той самий єдиний бот** `@OBVCarServiceWork_bot`, що й алерти інжесту.

Раніше в інжесті були PDF-звіти в Telegram + Google Sheets — видалено. Google Sheets вирізано повністю; PDF-звіти повернулись, але вже в **боті** (`src/bot/pdfReport.js`, `pdfkit`), а не в інжесті.

Пріоритети клієнта, які визначили архітектурні рішення: мінімізація регулярних витрат (усе на безкоштовних/копійчаних тарифах, де можливо) і надійність (жодні дані не повинні губитись мовчки).

## Команди

```
npm install

npm run poll                              # інжест: обробити нові дзвінки з моменту чекпоінта + ретрай pending-черги
npm run bot                               # запустити Telegram-бот звітності (long-polling; треба TELEGRAM_BOT_TOKEN)
npm run test:call -- <generalCallID>      # смок-тест одного дзвінка: Binotel record → OpenAI transcribe, без БД
npm run backfill -- "2026-07-01 00:00:00" "2026-07-03 23:59:59"   # обробити довільний історичний період
npm run reattribute:shared                # переприв'язати старі 901/902 дзвінки до людей (AI по збережених транскриптах)
npm run retranscribe:recent               # одноразово: останні 5 дзвінків кожного оператора-людини (менеджери+Богдан) перетранскрибувати через ElevenLabs (діаризація), оновити transcript
```

Тестів і лінтера в проєкті немає. Перевірка коректності — `node --check src/*.js` (синтаксис) + реальний прогін проти живих Binotel/OpenAI/Neon креденшлів з `.env` (усе покрито живими тестами по ходу розробки, автоматизованого test suite нема).

`JOB_TYPE=poll` — єдиний режим `src/jobs/index.js` (те, що задеплоєно). Історичний `report`-режим видалено разом зі звітністю.

Код на **ESM** (`import`/`export`, `"type": "module"` в package.json). Відносні імпорти — з розширенням `.js`. CJS-залежності підключаються через default-import (напр. `import pg from 'pg'; const { Pool } = pg;`).

## Структура репо

Монорепо, feature-based. Стандарт узгоджено під майбутній окремий бот звітності:

```
src/
  core/     — спільне ядро (використовують і jobs/, і bot/):
              store, binotel, transcribe, elevenlabs, classifyCall, identifyManager, telegram, retry
  jobs/     — задеплоєний cron-джоб (інжест): index.js (entrypoint) + pollNewCalls + processCalls
  scripts/  — одноразові інструменти: backfill, testSingleCall, reattributeShared, retranscribeRecent
  bot/      — Telegram-бот звітності (grammy, окремий токен, окремий сервіс):
              index.js (entrypoint) + keyboards + stats + archive + report + analyze + kb + time + ui
              (див. src/bot/README.md)
```

Коли пишеться нижче просто "processCalls.js", "store.js" тощо — шукати за цією мапою (`core/`, `jobs/` чи `bot/`).

## Архітектура

**Один cron-джоб** (pm2 `obv-poller`, `cron_restart: */15 * * * *`, `JOB_TYPE=poll`) робить усе:

1. **`retryPendingCalls()`** (`processCalls.js`) — спершу ретраїть усе, що застрягло в `pending_calls` з минулих разів
2. **Поллінг** (`pollNewCalls.js`): чекпоінт (`app_state.last_polled_until`) → зараз. Не фіксоване "останні N хв" — щоб пропущений/затриманий cron-запуск не створював діру. `listCallsForPeriod` (Binotel) обмежений 24 год на запит, тому діапазон ріжеться на шматки по 23 год (`processCalls.js: splitIntoChunks`)
3. Для кожного нового дзвінка: якщо `durationSec <= 0` — пропустити (не відповіли); якщо `recordingStatus != 'uploaded'` — в `pending_calls` на ретрай; інакше — `getCallRecordUrl` → `transcribeAudio` (див. «Транскрипція» нижче) → `classifyCall` (окремий легкий структурований запит: успішність / найслабший етап / оцінка 1-10) → **резолвинг імені оператора** (див. нижче) → `saveCall` в Postgres (транскрипт + класифікація + `manager_name` разом)

**Транскрипція (`core/transcribe.js` + `core/elevenlabs.js`).** **Первинний шлях — ElevenLabs STT (Scribe):** `POST /v1/speech-to-text` з `diarize:true` робить транскрипцію + **розділення мовців в одному виклику**. З `words[]` (кожне слово має `speaker_id`) збираємо репліки (`buildTurns`) і зберігаємо **готовий діалог «Менеджер:/Клієнт:»** прямо як `transcript`. **Хто саме менеджер (`pickManagerSpeaker`):** LLM аналізує ВЕСЬ діалог (не лише початок) із чітким двомовним промптом (маркери оператора: вітання від сервісу, марка авто, запис, ціни, майстер) + повертає reasoning і confidence; підстраховка — **детермінована keyword-евристика** `heuristicManager` (рахує оператор-маркери uk/ru на мовця): застосовується при фейлі LLM, поза-списковій відповіді, і як tie-break при low-confidence. Останній резерв — перший мовець. (Стара наївна «перший=менеджер» давала переставлені ролі, коли клієнт кидав перше «алло» — цей кейс тепер ловиться.) Тобто розділення мовців відбувається на інжесті, а не при відкритті архіву → архів миттєвий. **Fallback — OpenAI** (якщо нема `ELEVENLABS_API_KEY` / API впав / вичерпано кредити): стара логіка — транскрипція з укр-ухилом → детекція spoken-vs-written (суржик = uk) → перетранскрибація з примусовою мовою, якщо не збігається (фікс «українську записало як російську»). `CALL_LANGUAGE` (env) форсить мову для обох шляхів. Тариф ElevenLabs для ~50 дзв/день — **Scale** (STT = 330 кредитів/хв; ~1,5 млн кредитів/міс).

Ніякої звітності на кожному запуску вже немає — це кінець флоу інжесту.

### Бот звітності (`src/bot/`, окремий persistent-процес під pm2, long-polling grammy)

Персистентний процес (не cron), стартує `bot.start()`.

**Система ролей / контроль доступу (`bot/access.js` + `bot/roles.js`, таблиця `bot_users`).** Доступ більше НЕ керується env-allowlist'ом — ролі живуть у БД. Ролі — фіксований ENUM (без «додати роль»):
- **director** та **marketer** — однаковий ПОВНИЙ доступ (admin-tier): статистика всіх, архів, база знань (питання+файли), промпт аналізу, керування ролями.
- **manager** — «📊 Мій звіт» (стата ЛИШЕ по собі + свій телефон) і база знань (питання). Ідентичність — через `bot_users.operator_name` (директор лінкує при додаванні), тож у звіт по собі природно входять і дзвінки зі спільних номерів, атрибутовані на те саме ім'я.
- **mechanic** — лише база знань (питання).

`access.js`: `getUser` (роль по `telegram_id`, кеш із `invalidateRole`), `canAccess(role, feature)` + `featureOf(ctx)` — центральний gate в auth-middleware (класифікує кожен апдейт у фічу й пускає/блокує). `mainMenu(role)` і нативні команди (`setMyCommands` зі `scope:{type:'chat'}`) підбираються під роль. **Bootstrap:** `seedDirectors()` на старті сідить user-id з `TELEGRAM_BOOTSTRAP_CHAT_IDS` як директорів (щоб не залочитись), відсіюючи негативні (групові) id — ролі прив'язані до `ctx.from.id` (user).

**Керування ролями (`roles.js`, admin-only, блок «👥 Ролі» / `/roles`):** кнопки Маркетологи/Менеджери/Механіки (директорів UI навмисно не чіпає — захист seed-власника від видалення). Роль → список людей + «➕ Додати» → людина → «🗑 Видалити». **Додавання** (Telegram не дає ботам шукати за `@username` — це навмисне обмеження приватності): (1) `request_users` — обрати колегу з контактів, отримуємо `user_id`(+ім'я/username) МИТТЄВО, без потреби, щоб людина спершу писала боту (Bot API 7.2, grammy `Keyboard.requestUsers`); (2) поділитися контактом (є `user_id` → active, інакше pending за телефоном); (3) ввести телефон → pending. **Pending → active:** незнайомець ділиться СВОЇМ контактом на `/start`, `activatePendingByPhone` матчить за останніми 9 цифрами. Для manager після додавання — вибір оператора з roster (`getOperators`) → `operator_name`. ⚠️ Навіть маючи `user_id`, бот не може написати людині першим, поки вона не натисне Start (правило Telegram) — тож особисті авто-звіти неможливі; «Мій звіт» менеджер тягне сам інтерактивно.

**Навігація:** (1) нативні команди — кнопка «Menu» біля поля вводу, **список підбирається під роль** (`setMyCommands` зі `scope:{type:'chat'}`): admin — `/menu /stats /archive /ask /files /report /prompt /roles /settings`; manager — `/menu /myreport /ask`; mechanic — `/menu /ask`; (2) inline-меню в повідомленнях (`keyboards.js: mainMenu(role)`) + inline «« Меню» на всіх екранах. Постійну reply-клавіатуру (`quickKeyboard`/`QUICK`) **прибрано** (забагато UI-навантаження) — `/start` шле `remove_keyboard`, щоб очистити її в старих клієнтів. Нативний список команд свідомо ширший за inline-меню: **`/files` (Файли) і `/settings` (Налаштування) є ЛИШЕ в нативному списку**, inline-кнопок для них немає (свідомо, щоб не перевантажувати меню). Inline-кнопки admin-меню: «📊 Статистика менеджера», «🗂 Архів розмов», «📚 База знань» (`kb:ask`), «🔄 Звіт зараз», «👥 Ролі».

**Модель «активного екрана»** (`bot/ui.js: showScreen()`, замінила прямі `editMessageText`): меню/пікер завжди живе в НАЙнижчому повідомленні, щоб фокус юзера (низ чату) збігався з дією. Якщо натиснуте меню ще внизу — редагуємо на місці (плавно, без спаму); якщо його «поховало» під контентом (транскрипт / відповідь KB / текст промпту) — надсилаємо свіже меню внизу й видаляємо старе. `installMessageTracker` (API-transformer) веде id останнього надісланого ботом повідомлення на чат (щоб знати, чи меню ще внизу); `ctx.session.screenId` — id активного екрана (для видалення старого). Контентні дії (перегляд промпту тощо) після себе ре-планують меню вниз. Best practice підтверджено пошуком (стандарт — edit-in-place; тут адаптовано під контент-важкий бот).

Меню (inline-кнопки):

1. **Авто-звіти** (`report.js`) — власний планувальник (`setInterval` 30с звіряє київський час зі слотами). **Слоти часу — з `/settings` → «🕒 Час звітів» (`app_state.report_times`), не env** (`maybeSendScheduledReport` читає `getReportTimes()` щотіку; дефолт, поки не задано — `13:00,19:30`; порожній список = звітів нема). **Це НЕ окремий cron, а in-process інтервал усередині persistent-процесу бота** — на VPS достатньо тримати процес `node src/bot/index.js` живим (pm2 `obv-bot`). Період — з моменту попереднього авто-звіту (`app_state.last_report_slot`/`last_report_until`, дедуп слоту переживає рестарт; in-memory lock від подвійного спрацювання за хвилину). Групує дзвінки по менеджеру → OpenAI-резюме (`analyze.js`, промпт редагується через `/prompt`, див. фічу 5) → **PDF-файл** (`pdfReport.js`, `pdfkit`): одна сторінка на менеджера, заголовки-CAPS, `*жирний*`, булети, роздільники `---`. PDF будується один раз (`renderReport()`), тоді розсилається. **Кому надсилати авто-звіти — список отримувачів з `/settings` (`app_state.report_recipients`), не env** (`sendScheduledReport()` читає його на кожному слоті → `api.sendDocument` кожному; фейл одному не блокує інших). Кнопка «Звіт зараз» (`sendReport()`, ручний за сьогодні, стан розкладу не чіпає) шле PDF **тому, хто натиснув** (`ctx.chat.id`), а не в список. Формат PDF ідентичний скрізь.
2. **Статистика менеджера** (`stats.js`) — менеджер → період (день / тиждень з пн / місяць / квартал, київські, `bot/time.js`) → (а) **короткий числовий блок ЗАВЖДИ зверху** (к-сть/успішність/бал/найчастіший слабкий етап — з `getOperatorStats`, незалежно від промпту) + (б) **розгорнутий AI-аналіз за період** нижче: `getOperatorCallsWithTranscripts` → `analyze.js: analyzeManager` (той самий системний промпт із `/prompt`, що й у звітах) → `sendLong`. Кнопки нотаток (`manager_notes`). Той самий блок (а)+(б) — і в «Мій звіт» (`me:go`). Слабкі етапи в блоці (а) задаються в `core/classifyCall.js: WEAKEST_STAGES` (класифікація на інжесті; зміна діє лише на нові дзвінки). ⚠️ Аналіз генерується щоразу «брудно» (повний OpenAI-виклик, ~20-40с, без кешу) — див. «Активне нагадування» вгорі; кеш/оптимізація — окрема майбутня задача.
3. **Архів розмов** (`archive.js`) — менеджер → період → список (пагінація) → дзвінок: **розмова у форматі діалогу** + "🎧 Прослухати" (свіжий `getCallRecordUrl` з Binotel → `sendAudio`). Для нових дзвінків діалог «Менеджер:/Клієнт:» уже готовий (ElevenLabs-діаризація на інжесті) — показується **миттєво, без AI-запиту**. Для старих/fallback-транскриптів (суцільний текст, без міток; `looksDiarized()` це визначає) залишився запасний on-demand `bot/dialogue.js: formatDialogue` (OpenAI, ~10-20с, фейл → сирий текст). У кнопці дзвінка успішність позначена 👍/👎. У списку менеджерів біля імені — період активності `dd.mm.yy-dd.mm.yy` (`operatorListKeyboard(..., {showDates:true})`). У кнопці дзвінка успішність позначена 👍/👎 (не ✅/❌). У списку менеджерів біля імені показується період активності `dd.mm.yy-dd.mm.yy` (`operatorListKeyboard(..., {showDates:true})`): від найранішого обробленого дзвінка (`getOperators` → `MIN(start_time)`, проксі «коли Binotel уперше побачив ім'я» — реальної дати реєстрації Binotel API не віддає) до сьогодні. Тільки в архіві; у статистиці той самий пікер без дат.

**Аліаси імен операторів** (`bot/operators.js: displayName`) — мапа «сирий `manager_name` → людське ім'я», застосовується ЛИШЕ на рендері (кнопки, заголовки, деталі дзвінка, PDF-звіт). БД і callback-data лишаються сирими → групування/запити/атрибуція не змінюються, мапа повністю оборотна. Дефолт: номер директора `0674738200` → «Богдан» (показується як `📱 Богдан`, а не `☎️ Спільний 0674738200`). Розширюється env `OPERATOR_ALIASES` (пари `номер=Ім'я` через кому, мерджиться поверх дефолту). **Емоджі в пікері операторів** (`keyboards.js: operatorLabel`): іменований/аліасований оператор — `📱` (клієнт просив показувати як «персональний номер»); лише неатрибутований спільний номер лишається `☎️ Спільний`.
4. **База знань** (`kb.js`, кнопка «📚 База знань» → `kb:ask`, команда `/ask`) — **RAG** по завантажених посібниках. Власник надсилає боту документ (PDF `unpdf` / DOCX `mammoth` / TXT) → **бот питає, ДЛЯ КОГО файл** (категорія-audience: 🔧 Механікам / 💼 Менеджерам / 👥 Обом; файл стешиться в `ctx.session.pendingKbDoc`, інжест відкладений до вибору `kb:aud:*`) → **посторінковий витяг** (`extractPages`: PDF `mergePages:false` → текст на кожну сторінку; DOCX/TXT — сторінок нема, `page=null`) → **page-aware нарізка** (`chunkDocument`, ~2400 симв., трекає діапазон сторінок на чанк) → embeddings (`text-embedding-3-small`) → Postgres (`kb_docs.audience` + `kb_chunks` з `vector(1536)` + `page_start/page_end`, **pgvector**).
   - **Пошук (покращений, multi-query):** `retrieve()` — `expandQueries` просить модель дати 3 переформулювання питання, embed'ить оригінал+варіанти, шукає кожен (`searchKbChunks`, по 8), мерджить за `chunkId` (найкраща відстань), бере топ-8. Це полагодило слабку повторюваність («З чого складається двигун» vs «Як працює двигун»). Розширення — best-effort (падіння → лише сире питання).
   - **Категорії/доступ (audience):** `searchKbChunks(emb, k, audiencesForRole(role))`: `manager` → `manager`+`both`, `mechanic` → `mechanic`+`both`, `director`/`marketer` (admin) → усе (фільтр `null`). Механік не отримає посібник продажів і навпаки.
   - **Відповідь (структурована, `ANSWER_SCHEMA`):** модель повертає `{answer, usedSources[], usedGeneralKnowledge}`. **Дозволено ~10% загальних знань:** якщо фрагменти покривають лише частково — модель доповнює достовірними загальними знаннями (не галюцинуючи, не суперечачи посібникам) і ставить `usedGeneralKnowledge=true`. Відповідь — простим текстом (модель НЕ додає джерела сама).
   - **Блок джерел (детермінований, ОБОВ'ЯЗКОВИЙ):** будується в КОДІ з `usedSources` (`buildSourcesFooter`), а не моделлю. Групується по файлу, сторінки мерджаться (`pagesLabel` → `стор. 3–4, 12`). Назва файлу — **гіперпосилання-діплінк** `https://t.me/<bot>?start=kbdoc_<docId>` (клік → `/start kbdoc_<id>` у `index.js` → `openKbDocById` пересилає оригінал по `file_id`, з перевіркою audience). Відповідь шлеться **HTML** (`sendLong(..., {parseMode:'HTML'})`) — назви з `_` не ламають розмітку. Якщо додано загальні знання — окремий рядок-примітка.
   - Екран «📚 Файли»: у списку біля кожного файлу — категорія; деталі → 📄 Відкрити оригінал / **🔁 Змінити для кого** (`kb:audset`→`kb:audput`) / 🗑 Видалити (каскад) + «➕ Завантажити новий». `migrateKb()` guard'иться: нема pgvector → KB вимкнено, решта бота живе. **KB-екрани меню — простий текст без Markdown** (назви файлів з `_`); а от сама ВІДПОВІДЬ на питання — HTML (для діплінків). Після відповіді бот **лишається в режимі питань** (`awaiting=kb_question`); вихід — `/menu`. ⚠️ Сторінки показуються лише для файлів, залитих ПІСЛЯ цього оновлення (старі чанки мають `page_start=NULL`).
5. **Промпт аналізу** (`prompt.js`, команда `/prompt`) — власник переглядає / змінює / скидає системний промпт, яким AI оцінює роботу операторів у звітах. Меню: 👁 переглянути поточний, ✏️ змінити (надіслати новий текст повідомленням → `awaiting=prompt`), ↩️ скинути до стандартного. Зберігається в `app_state.analyze_prompt`; `analyze.js: getAnalyzePrompt()` повертає його або `DEFAULT_ANALYZE_PROMPT`. Змінений промпт одразу діє і в авто-звітах, і в «Звіт зараз». **`DEFAULT_ANALYZE_PROMPT` — структурований звіт під PDF** (розділи CAPS-заголовками: загальна характеристика / сильні сторони / системні помилки / найслабший етап / динаміка / топ-3 точки росту / готові формулювання; `*жирний*` для назв пунктів). Формат навмисно узгоджений із рендером `pdfReport.js` (CAPS-рядок → заголовок, `*...*` → bold, `-`/`•` → булет, `---` → роздільник; markdown нормалізується — `**подвійні**` зірочки не лишають артефактів).

6. **Налаштування** (`settings.js`, команда `/settings` — **лише нативна, БЕЗ inline-кнопки**, admin-only) — керування двома списками отримувачів (обидва в `app_state` як JSON `{id,name}[]`, тому повідомлення розсилається кільком людям одразу): **«⚠️ Сповіщення про поломки»** (`alert_recipients`, куди `core/telegram.js: sendAlert` шле алерти інжесту) і **«📊 Щоденні звіти»** (`report_recipients`, куди йдуть авто-звіти). На екрані списку: кнопка `🗑 <ім'я>` на кожного (видалення одним тапом), «➕ Додати» → reply-клавіатура (`request_users` «Обрати з контактів» / «Поділитися контактом» / надіслати числовий ID чату — останнє для груп на кшталт `-5426146652`). Отримувач = Telegram chat_id (юзер, що натиснув Start, або група). Store-обгортки — `getRecipients/addRecipient/removeRecipient(kind)`. Крім двох списків отримувачів, тут же **«🕒 Час звітів»** (`app_state.report_times`, `getReportTimes/addReportTime/removeReportTime`): київські слоти щоденних звітів, кероване кнопковим пікером **година → хвилина (крок 10 хв)** — формат завжди канонічний `HH:MM`, ввести невалідне неможливо. Слотів може бути скільки завгодно; порожньо = звітів нема; дефолт (поки не задано) — `13:00,19:30`. Callback'и: `set:times` (список), `set:timeadd`→`set:timeh:<h>`→`set:timem:<h>:<m>` (додати), `set:timedel:<HHMM>` (прибрати). Gate — `featureOf` повертає `settings` для `set*` / `/settings` (admin-only). ⚠️ Замінило env-змінні `TELEGRAM_CHAT_ID`, `BOT_REPORT_CHAT_ID` і `BOT_REPORT_TIMES` (усі прибрані). Хендлери `message:contact`/`message:users_shared` у `roles.js` і `settings.js` пропускають чужий апдейт далі через `next()` — ланцюжок: roles → settings → save_phone (в `index.js`).

Бот НЕ пише в `calls`/`pending_calls` — лише читає; пише в `manager_notes`, `kb_docs`/`kb_chunks`, `bot_users` (ролі) і ключі `app_state` (звітні + `analyze_prompt` + `alert_recipients`/`report_recipients`).

### Ідентифікація оператора (`resolveManagerName` в `processCalls.js`)

**Джерело правди — Binotel, а не локальна таблиця.** Таблиці `managers` більше немає (клієнт керує операторами в Binotel). Оператор ідентифікується **іменем**, і по цьому ж імені (`manager_name`) далі групується вся статистика/архів/звіти.

- **Особистий номер** (903/904/905) → `manager_name` = `employeeData.name` від Binotel (напр. "Роман", "Владимир", "Андрей" — саме так, як пише Binotel).
- **Спільна слухавка** (номер у `SHARED_EXTENSIONS`, дефолт `901,902`) або номер без імені → Binotel не знає, хто відповів, тому оператор визначається **з транскрипту**: `identifyManager.js` окремим OpenAI-запитом обирає, ХТО зі списку відомих операторів представився. Список кандидатів (**roster**) — динамічний, `getOperatorRoster()` = distinct не-числові `manager_name` з `calls` (тобто імена, які Binotel уже давав на особистих номерах). AI обмежений `enum`'ом кандидатів, тож повертає точно одне з тих імен (сам зводить "Володя"→"Владимир") або `null`. Збіг → це ім'я. Нема збігу/імені → лишається голий номер (`901`) як мітка.

Тобто roster сам «наповнюється» з Binotel: як тільки менеджер зробив дзвінок з особистого номера, його ім'я стає кандидатом для матчингу спільних дзвінків. Жодної ручної таблиці/aliases. **Клієнт попросить операторів представлятись** на спільних — від цього залежить якість атрибуції 901/902.

`reattributeShared.js` (`npm run reattribute:shared`) — переганяє вже збережені спільні дзвінки (з голим номером) через ту саму AI-ідентифікацію по збережених транскриптах.

### Надійність (свідомі архітектурні рішення, не міняти без причини)

- **Чекпоінт замість вікна** — гарантує відсутність дір при збоях cron
- **`pending_calls`** — дзвінок, що впав на будь-якому кроці, ретраїться на кожному наступному `poll` (не втрачається), до `MAX_PENDING_ATTEMPTS` (20), потім `status='failed'` + Telegram-алерт, рядок лишається в таблиці для ручної перевірки (нічого не видаляється мовчки)
- **`withRetry`** (`retry.js`) — обгортає кожен зовнішній виклик (Binotel/OpenAI/Telegram), 2-3 спроби з наростаючою паузою
- **Telegram (інжест) — тільки алерти.** `core/telegram.js` лише `sendMessage`/`sendAlert` (вихідні), для сповіщень про збої й невідомих менеджерів. `sendMessage` ділить текст на шматки ≤4096 і має markdown→plain fallback. Шле через той самий єдиний бот `@OBVCarServiceWork_bot` (вихідні виклики не конфліктують з `getUpdates` бота звітності).

### Схема БД (Postgres/Neon, `core/store.js: migrate()`)

- **`calls`** — `general_call_id` (унікальний), `internal_number`, `manager_name` (ім'я оператора — ключ групування скрізь), `start_time`, `duration_sec`, `transcript`, `is_success`, `weakest_stage`, `communication_score`. (Таблиці `managers`, колонок `manager_id` і `call_type` більше немає. Binotel є джерелом операторів; `call_type` (напрямок вх/вих) прибрано на прохання клієнта. `migrate()` містить ідемпотентне прибирання цих legacy-артефактів — `DROP COLUMN IF EXISTS manager_id`/`call_type` у `calls`, `DROP COLUMN IF EXISTS call_type` у `pending_calls`, `DROP TABLE IF EXISTS managers`; діє на всі рядки, нові й старі.)
- **`pending_calls`** — підмножина полів calls + `attempts`, `status` (`pending`/`failed`), `last_error`
- **`manager_notes`** — `id`, `operator_name` (ім'я оператора = `calls.manager_name`), `author`, `note`, `created_at`. Нотатки, які лишає власник у боті (фіча 2)
- **`bot_users`** — контроль доступу (ролі): `id`, `telegram_id` (UNIQUE, NULL поки invite pending), `role` (director|marketer|manager|mechanic), `phone`, `username`, `display_name`, `operator_name` (лінк менеджера до `calls.manager_name` для стати по собі), `status` (active|pending), `added_by`, `created_at`. **Це НЕ воскресіння старої `managers`** (та була про атрибуцію дзвінків) — тут суто авторизація «хто і що може в боті»; Binotel лишається джерелом правди щодо того, хто говорив. Функції в `store.js`: `getBotUser`/`getBotUsersByRole`/`upsertBotUserByTelegram`/`addPendingBotUser`/`activatePendingByPhone`/`setBotUserOperator`/`setBotUserPhone`/`deleteBotUser`/`seedDirector`
- **`kb_docs`** / **`kb_chunks`** — база знань (фіча 4): `kb_docs` (файл + `file_id`/`mime` для повторної відправки оригіналу + `chunk_count` + **`audience`** TEXT `mechanic`|`manager`|`both` — для кого посібник; `NOT NULL DEFAULT 'mechanic'`, що й бекфілить наявні рядки на механіка), `kb_chunks` (`doc_id` FK ON DELETE CASCADE, `ord`, `content`, `embedding vector(1536)`, hnsw-індекс cosine, **`page_start`/`page_end`** — діапазон сторінок чанка для цитування джерел; NULL для DOCX/TXT і для чанків, залитих до цього оновлення). Створюються окремо в `migrateKb()` (потребує `CREATE EXTENSION vector`), не в `migrate()`. Пошук фільтрує за `audience` залежно від ролі того, хто питає (`searchKbChunks(..., audiences)`)
- **`app_state`** — key-value: `last_polled_until` (чекпоінт інжесту); `last_report_slot`/`last_report_until` (розклад авто-звітів, пише **бот**); `analyze_prompt` (кастомний промпт аналізу, якщо власник змінив через `/prompt`; нема ключа → `DEFAULT_ANALYZE_PROMPT` з `analyze.js`); `alert_recipients`/`report_recipients` (JSON `{id,name}[]` — куди слати алерти інжесту й авто-звіти); `report_times` (JSON `["HH:MM"]` — київські слоти щоденних звітів; усе це керується в `/settings`, див. фічу 6). Обгортки — `store.js: getStoredAnalyzePrompt/…`, `getRecipients/addRecipient/removeRecipient(kind)`, `getReportTimes/addReportTime/removeReportTime` (як `getCheckpoint`/`getReportSlot`)

Час: інжест рахує все в UTC (чекпоінт — абсолютний момент), йому Kyiv не треба. **Боту** Kyiv потрібен (слоти звітів + межі періодів) — `src/bot/time.js` рахує через `Intl.DateTimeFormat`, без хардкод-офсету (ламається на EET/EEST). У `core/` спільного `time.js` немає — це суто ботова логіка.

## Env-змінні

Повний список і коментарі — `.env.example`. Ключове:
- `BINOTEL_API_KEY/SECRET`, `BINOTEL_BASE_URL` — ендпоінти підтверджені емпірично на реальному акаунті 2026-07-06 (`stats/list-of-calls-for-period.json` з `startTime`/`stopTime` в unix-секундах, `stats/call-record.json`)
- `OPENAI_ANALYZE_MODEL` — `classifyCall`, `identifyManager`, звіти, відповіді бази знань, лейблінг мовців (дефолт `gpt-4o-mini`). `OPENAI_EMBED_MODEL` — embeddings бази знань (дефолт `text-embedding-3-small`, 1536-dim). `OPENAI_TRANSCRIBE_MODEL` — лише для fallback-транскрипції (дефолт `gpt-4o-mini-transcribe`)
- `ELEVENLABS_API_KEY` — **первинний транскрайбер** (STT + діаризація в одному виклику, `core/elevenlabs.js`). Порожньо → fallback на OpenAI-транскрипцію (без діаризації). `ELEVENLABS_STT_MODEL` (дефолт `scribe_v1`), `ELEVENLABS_NUM_SPEAKERS` (дефолт `2` — телефон=2 сторони). Заголовок авторизації — `xi-api-key`. Тариф для ~50 дзв/день — **Scale** (STT 330 кредитів/хв ≈ 1,5 млн кредитів/міс; ~$330/міс). `CALL_LANGUAGE` форсить `language_code`
- `SHARED_EXTENSIONS` — comma-separated внутрішні номери спільних слухавок (дефолт `901,902`), для яких менеджер визначається з транскрипту
- `OPERATOR_ALIASES` — (бот, опційно) пари `номер=Ім'я` через кому для показу сирого `manager_name` під людським іменем (напр. `0674738200=Богдан`). Мерджиться поверх дефолту в `bot/operators.js` (де `0674738200=Богдан` уже прошито). Тільки відображення — БД не змінюється
- `DATABASE_URL` — Neon Postgres
- `TELEGRAM_BOT_TOKEN` — **єдиний бот на все**: `@OBVCarServiceWork_bot` (id 8683695884, новий, чистий, без webhook — перевірено `getWebhookInfo` 2026-07-10). Обслуговує і бота звітності (`getUpdates`), і вихідні алерти інжесту (`sendMessage`) — це не конфліктує. `@obvcarservicebot` (id 8979148940) більше **НЕ** використовується (його webhook належить CarBook — `api.carbook.pro/api/telegram/webhook/7971`)
- **Єдиний Telegram-chat-id у env — `TELEGRAM_BOOTSTRAP_CHAT_IDS`** (раніше `BOT_ALLOWED_CHAT_IDS`; перейменовано). Це **більше НЕ allowlist доступу**, а лише **bootstrap-сід директорів** на старті (`seedDirectors()` заносить ці user-id як `director` у `bot_users`, щоб не залочитись на чистій БД). Після першого запуску доступ керується через блок «Ролі» в боті; env-змінну можна не чіпати. Негативні (групові) id ігноруються при сідінгу.
- **Куди й КОЛИ йдуть алерти й авто-звіти — БІЛЬШЕ НЕ в env**, а в `/settings` (див. фічу 6): списки `app_state.alert_recipients` / `report_recipients` + слоти `app_state.report_times`. Прибрані env-змінні: `TELEGRAM_CHAT_ID` (був алерт-чат-ГРУПА `-5426146652`), `BOT_REPORT_CHAT_ID` (був звітний user `429132087`) і `BOT_REPORT_TIMES` (був `13:00,19:30`, тепер дефолт у коді, поки директор не задав слоти). (`TELEGRAM_ADMIN_CHAT_ID` — прибрано ще раніше.) На чистій БД списки отримувачів порожні — admin додає у боті (для групи — опція «числовий ID»); час звітів до редагування = дефолт `13:00,19:30`.
- `OPERATOR_ALIASES` — (бот, опційно) пари `номер=Ім'я` для показу сирого `manager_name` під людським іменем (див. вище)

## Деплой

Детальна інструкція — `DEPLOY.md`. Коротко: VPS на **AlmaLinux 10**, обидва процеси тримає **pm2** (`ecosystem.config.cjs` у корені репо). **Два процеси з одного репо**: `obv-poller` (`node src/jobs/index.js`, pm2 `cron_restart: */15 * * * *`, `JOB_TYPE=poll`) і `obv-bot` (`node src/bot/index.js`, `autorestart`, постійно online). Обидва читають один `.env` з кореня. Автозапуск після ребуту — `pm2 save && pm2 startup`. Оновлення коду — `git pull && npm install --omit=dev && pm2 reload ecosystem.config.cjs`.

- **Ідентифікація 901/902** — ✅ обов'язкова, лишилась: `identifyManager.js` (enum-вибір з roster). Якість залежить від того, чи оператори представляються (клієнт це організує)
- **Telegram-бот звітності** — ✅ реалізовано (`src/bot/`, grammy, єдиний бот `@OBVCarServiceWork_bot` через `TELEGRAM_BOT_TOKEN`): авто-звіти 13:00/19:30, статистика менеджера (день/тиждень/місяць/квартал) + нотатки, архів розмов з прослуховуванням аудіо. Хостинг — VPS (AlmaLinux) під pm2, persistent-процес `obv-bot`
- **База знань (RAG)** — ✅ реалізовано (`bot/kb.js` + pgvector): завантаження PDF/DOCX/TXT боту, embeddings у Postgres, кнопка «📚 База знань» (`kb:ask`, `/ask`) → відповідь лише з посібників. Протестовано наскрізь 2026-07-10 (Neon має pgvector 0.8.1). Скановані PDF/зображення без текстового шару → потрібен OCR (не реалізовано). Клієнт дасть реальні файли — перевірити витяг тексту з його PDF
- **Категорії бази знань (audience)** — ✅ реалізовано: у кожного файлу є категорія `mechanic`/`manager`/`both` (`kb_docs.audience`). При аплоуді директор/маркетолог обирає, для кого файл; наявні файли бекфілнуто на `mechanic`. Відповіді фільтруються за роллю того, хто питає (`audiencesForRole`): механік не бачить менеджерських посібників і навпаки, admin — усе. Категорію можна змінити на екрані файлу («🔁 Змінити для кого»)
- **Розгорнутий аналіз у статистиці** — ✅ реалізовано: «Статистика менеджера» і «Мій звіт» показують короткий числовий блок зверху + AI-аналіз за період (той самий промпт із `/prompt`). ⚠️ Генерується щоразу «брудно» (OpenAI ~20-40с, без кешу) — кеш/швидкодія заплановані окремою задачею (див. «Активне нагадування» вгорі)
- **Джерела + пошук у базі знань** — ✅ реалізовано: (1) **обов'язковий блок джерел** унизу відповіді, побудований у коді з реально використаних фрагментів — файл(и) з номерами сторінок і **гіперпосиланням-діплінком** на бота (`?start=kbdoc_<id>` → пересилання оригіналу); (2) **покращений пошук** через multi-query (модель дає переформулювання, мерджимо результати) — вирішує «однакове питання різними словами»; (3) модель може **докинути ~10% загальних знань** (позначається окремо), коли посібники не покривають питання; (4) сторінки в `kb_chunks.page_start/end` (лише для нових файлів). Архітектурне рішення: pgvector лишається (пошук), лінк на файл — через Telegram-діплінк, без зовнішнього хостингу. ⚠️ Справжній перехід на конкретну сторінку PDF Telegram не вміє — сторінки вказано текстом, лінк відкриває весь файл
- **Транскрипція через ElevenLabs (STT + діаризація)** — ✅ реалізовано: первинний шлях — ElevenLabs Scribe (`core/elevenlabs.js`), який в одному виклику дає текст + розділення мовців; збираємо готовий діалог «Менеджер:/Клієнт:» (роль визначає дешевий gpt-4o-mini, фейл → «перший=менеджер») і зберігаємо як `transcript`. Розділення мовців якісніше за старий пост-факт OpenAI-підхід, і архів став миттєвим. OpenAI лишився **fallback**'ом (нема ключа/впав/кредити). Тариф для ~50 дзв/день — Scale ($330/міс). ⚠️ Перевірити наживо на реальних дзвінках клієнта (якість діаризації, uk/ru autodetect); за потреби — форс `CALL_LANGUAGE=uk` або `ELEVENLABS_STT_MODEL=scribe_v2`
- **Визначення «хто менеджер» у діаризації** — ⚠️ частково: діаризація ElevenLabs якісна, але призначення ролей Менеджер/Клієнт було ненадійним (плуталось). Покращено (2026-07-22): LLM по всьому діалогу + reasoning/confidence + детермінована keyword-евристика (`heuristicManager`, uk/ru оператор-маркери) як fallback і tie-break; наївне «перший=менеджер» прибрано. Аудіо **моно** (підтверджено), тому канальний спосіб недоступний. **Заплановано (макс. надійність): голосовий відбиток** — енролмент голосу менеджера + порівняння голосових ембедингів мовців (потрібен окремий локальний Python-сервіс на VPS: pyannote/SpeechBrain). Робити, якщо покращеного LLM+евристики буде замало
