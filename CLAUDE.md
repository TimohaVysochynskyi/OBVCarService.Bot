## Що це за проєкт

Автоматизація для клієнта (СТО/автосервіс): дзвінки менеджерів через Binotel (АТС) автоматично транскрибуються через OpenAI, класифікуються (успішність / найслабший етап / оцінка) і **зберігаються в Postgres із прив'язкою до конкретного менеджера**; окремий Telegram-бот читає ту саму БД і віддає звіти/статистику/архів. Раніше було на n8n; переписано на чистий Node.js, деплой — два Render-сервіси (cron-інжест + persistent-бот).

**Два компоненти (монорепо):**
1. **Інжест** (`src/jobs/`, Render Cron Job) — збір, транскрипція, класифікація і атрибуція дзвінків у БД. Звітів тут немає свідомо.
2. **Бот звітності** (`src/bot/`, Render Background Worker) — інтерактивний Telegram-бот (grammy, long-polling): авто-звіти, статистика менеджера, архів розмов, база знань (каркас). Читає ту саму БД. **Той самий єдиний бот** `@OBVCarServiceWork_bot`, що й алерти інжесту.

Раніше в інжесті були PDF-звіти в Telegram + Google Sheets — видалено (переїхало логічно в бот, але вже текстом, без PDF/Sheets).

Пріоритети клієнта, які визначили архітектурні рішення: мінімізація регулярних витрат (усе на безкоштовних/копійчаних тарифах, де можливо) і надійність (жодні дані не повинні губитись мовчки).

## Команди

```
npm install

npm run poll                              # інжест: обробити нові дзвінки з моменту чекпоінта + ретрай pending-черги
npm run bot                               # запустити Telegram-бот звітності (long-polling; треба TELEGRAM_BOT_TOKEN)
npm run test:call -- <generalCallID>      # смок-тест одного дзвінка: Binotel record → OpenAI transcribe, без БД
npm run backfill -- "2026-07-01 00:00:00" "2026-07-03 23:59:59"   # обробити довільний історичний період
npm run reattribute:shared                # переприв'язати старі 901/902 дзвінки до людей (AI по збережених транскриптах)
```

Тестів і лінтера в проєкті немає. Перевірка коректності — `node --check src/*.js` (синтаксис) + реальний прогін проти живих Binotel/OpenAI/Neon креденшлів з `.env` (усе покрито живими тестами по ходу розробки, автоматизованого test suite нема).

`JOB_TYPE=poll` — єдиний режим `src/jobs/index.js` (те, що задеплоєно на Render). Історичний `report`-режим видалено разом зі звітністю.

Код на **ESM** (`import`/`export`, `"type": "module"` в package.json). Відносні імпорти — з розширенням `.js`. CJS-залежності підключаються через default-import (напр. `import pg from 'pg'; const { Pool } = pg;`).

## Структура репо

Монорепо, feature-based. Стандарт узгоджено під майбутній окремий бот звітності:

```
src/

  scripts/  — одноразові інструменти: backfill, testSingleCall, reattributeShared
```

Коли пишеться нижче просто "processCalls.js", "store.js" тощо — шукати за цією мапою (`core/`, `jobs/` чи `bot/`).

## Архітектура

**Один Render Cron Job** (`binotel-poller`, `*/15 * * * *`, `JOB_TYPE=poll`) робить усе:

1. **`retryPendingCalls()`** (`processCalls.js`) — спершу ретраїть усе, що застрягло в `pending_calls` з минулих разів
2. **Поллінг** (`pollNewCalls.js`): чекпоінт (`app_state.last_polled_until`) → зараз. Не фіксоване "останні N хв" — щоб пропущений/затриманий cron-запуск не створював діру. `listCallsForPeriod` (Binotel) обмежений 24 год на запит, тому діапазон ріжеться на шматки по 23 год (`processCalls.js: splitIntoChunks`)
3. Для кожного нового дзвінка: якщо `durationSec <= 0` — пропустити (не відповіли); якщо `recordingStatus != 'uploaded'` — в `pending_calls` на ретрай; інакше — `getCallRecordUrl` → `transcribeAudio` (OpenAI, див. «Мова» нижче) → `classifyCall` (окремий легкий структурований запит: успішність / найслабший етап / оцінка 1-10) → **резолвинг імені оператора** (див. нижче) → `saveCall` в Postgres (транскрипт + класифікація + `manager_name` разом)

**Мова транскрибації (`core/transcribe.js`).** Проблема: українська розмовна з суржиком («да», «шо», «тіки») часто транскрибується як російська. Флоу: (1) транскрипція з українським prompt-ухилом; (2) окремий OpenAI-запит визначає `spoken` (реально сказана мова; суржик = **uk**) і `transcriptLanguage` (якою мовою вийшов текст); (3) якщо не збігаються (сказали uk — записало ru) → **переписуємо з примусовою `language`**. `CALL_LANGUAGE` (env) форсить мову й вимикає детекцію. Підтверджено наживо 2026-07-10: ru-дзвінок лишається ru, а uk-записаний-як-ru переписується на uk.

Ніякої звітності на кожному запуску вже немає — це кінець флоу інжесту.

### Бот звітності (`src/bot/`, окремий Render Background Worker, long-polling grammy)

Персистентний процес (не cron), стартує `bot.start()`. Owner-only: allowlist `BOT_ALLOWED_CHAT_IDS` (порожній → бот показує твій ID і нікого не пускає).

**Навігація (щоб меню не губилось):** (1) постійна reply-клавіатура над полем вводу — `📊 Статистика / 🗂 Архів / 🔄 Звіт зараз / ☰ Меню` (кнопки шлють текст-мітку, `index.js` message:text її роутить, працює навіть посеред вводу нотатки); (2) нативні команди `/menu /stats /archive /report` (`setMyCommands` + `setChatMenuButton` → кнопка «Menu» біля вводу); (3) inline «« Меню» на всіх екранах. `keyboards.js: quickKeyboard()` + `QUICK`.

Меню (inline-кнопки):

1. **Авто-звіти** (`report.js`) — власний планувальник (`setInterval` 30с звіряє київський час зі слотами `BOT_REPORT_TIMES`, дефолт `13:00,19:30`). Період — з моменту попереднього авто-звіту (`app_state.last_report_slot`/`last_report_until`, дедуп слоту переживає рестарт; in-memory lock від подвійного спрацювання за хвилину). Групує дзвінки по менеджеру → OpenAI-резюме (`analyze.js`, промпт-заглушка) → одне текстове повідомлення (chunk >4096). Кнопка "Звіт зараз" — ручний за сьогодні, стан розкладу не чіпає.
2. **Статистика менеджера** (`stats.js`) — менеджер → період (день / тиждень з пн / місяць / квартал, київські, `bot/time.js`) → к-сть/успішність/бал/слабкий етап + нотатки (`manager_notes`).
3. **Архів розмов** (`archive.js`) — менеджер → період → список (пагінація) → дзвінок: транскрипт + "🎧 Прослухати" (свіжий `getCallRecordUrl` з Binotel → `sendAudio`).
4. **База знань / «Поставити питання»** (`kb.js`) — **RAG** по завантажених посібниках. Власник надсилає боту документ (PDF `unpdf` / DOCX `mammoth` / TXT) → витяг тексту → нарізка (`chunkText`, ~2400 симв. з перекриттям) → embeddings (`text-embedding-3-small`) → зберігання в Postgres (`kb_docs` + `kb_chunks` з `vector(1536)`, **pgvector**). Питання: embedding → cosine-пошук топ-6 шматків (`<=>`) → gpt-4o-mini відповідає **лише** з тих фрагментів (інакше «немає відповіді»), з посиланням на файл. Екран «📚 Файли»: кнопка на кожен файл → деталі (📄 Відкрити оригінал через збережений Telegram `file_id` / 🗑 Видалити з підтвердженням, каскад) + «➕ Завантажити новий». `migrateKb()` guard'иться: нема pgvector → KB вимкнено, решта бота живе. **Усі KB-екрани — простий текст без Markdown** (назви файлів часто містять `_`, що ламало парсинг Markdown — був баг «нічого не відбувається»).

Бот НЕ пише в `calls`/`pending_calls` — лише читає; пише в `manager_notes`, `kb_docs`/`kb_chunks` і звітні ключі `app_state`.

### Ідентифікація оператора (`resolveManagerName` в `processCalls.js`)

**Джерело правди — Binotel, а не локальна таблиця.** Таблиці `managers` більше немає (клієнт керує операторами в Binotel). Оператор ідентифікується **іменем**, і по цьому ж імені (`manager_name`) далі групується вся статистика/архів/звіти.

- **Особистий номер** (903/904/905) → `manager_name` = `employeeData.name` від Binotel (напр. "Роман", "Владимир", "Андрей" — саме так, як пише Binotel).
- **Спільна слухавка** (номер у `SHARED_EXTENSIONS`, дефолт `901,902`) або номер без імені → Binotel не знає, хто відповів, тому оператор визначається **з транскрипту**: `identifyManager.js` окремим OpenAI-запитом обирає, ХТО зі списку відомих операторів представився. Список кандидатів (**roster**) — динамічний, `getOperatorRoster()` = distinct не-числові `manager_name` з `calls` (тобто імена, які Binotel уже давав на особистих номерах). AI обмежений `enum`'ом кандидатів, тож повертає точно одне з тих імен (сам зводить "Володя"→"Владимир") або `null`. Збіг → це ім'я. Нема збігу/імені → лишається голий номер (`901`) як мітка.

Тобто roster сам «наповнюється» з Binotel: як тільки менеджер зробив дзвінок з особистого номера, його ім'я стає кандидатом для матчингу спільних дзвінків. Жодної ручної таблиці/aliases. **Клієнт попросить операторів представлятись** на спільних — від цього залежить якість атрибуції 901/902.

`reattributeShared.js` (`npm run reattribute:shared`) — переганяє вже збережені спільні дзвінки (з голим номером) через ту саму AI-ідентифікацію по збережених транскриптах.

### Надійність (свідомі архітектурні рішення, не міняти без причини)

- **Чекпоінт замість вікна** — гарантує відсутність дір при збоях cron
- **`pending_calls`** — дзвінок, що впав на будь-якому кроці, ретраїться на кожному наступному `poll` (не втрачається), до `MAX_PENDING_ATTEMPTS` (20), потім `status='failed'` + Telegram-алерт, рядок лишається в таблиці для ручної перевірки (нічого не видаляється мовчки)
- **`withRetry`** (`retry.js`) — обгортає кожен зовнішній виклик (Binotel/OpenAI/Telegram), 2-3 спроби з наростаючою паузою
- **Telegram (інжест) — тільки алерти.** `core/telegram.js` лише `sendMessage`/`sendAlert` (вихідні), для сповіщень про збої й невідомих менеджерів. `sendMessage` ділить текст на шматки ≤4096 і має markdown→plain fallback. Шле через той самий єдиний бот `@OBVCarServiceWork_bot` (вихідні виклики не конфліктують з `getUpdates` бота звітності).

### Схема БД (Postgres/Neon, `core/store.js: migrate()`)

- **`calls`** — `general_call_id` (унікальний), `internal_number`, `manager_name` (ім'я оператора — ключ групування скрізь), `call_type`, `start_time`, `duration_sec`, `transcript`, `is_success`, `weakest_stage`, `communication_score`. (Таблиці `managers` й колонки `manager_id` більше немає — Binotel є джерелом операторів.)
- **`pending_calls`** — підмножина полів calls + `attempts`, `status` (`pending`/`failed`), `last_error`
- **`manager_notes`** — `id`, `operator_name` (ім'я оператора = `calls.manager_name`), `author`, `note`, `created_at`. Нотатки, які лишає власник у боті (фіча 2)
- **`kb_docs`** / **`kb_chunks`** — база знань (фіча 4): `kb_docs` (файл + `file_id`/`mime` для повторної відправки оригіналу + `chunk_count`), `kb_chunks` (`doc_id` FK ON DELETE CASCADE, `ord`, `content`, `embedding vector(1536)`, hnsw-індекс cosine). Створюються окремо в `migrateKb()` (потребує `CREATE EXTENSION vector`), не в `migrate()`
- **`app_state`** — key-value: `last_polled_until` (чекпоінт інжесту); `last_report_slot`/`last_report_until` (розклад авто-звітів, пише **бот**)

Час: інжест рахує все в UTC (чекпоінт — абсолютний момент), йому Kyiv не треба. **Боту** Kyiv потрібен (слоти звітів + межі періодів) — `src/bot/time.js` рахує через `Intl.DateTimeFormat`, без хардкод-офсету (ламається на EET/EEST). У `core/` спільного `time.js` немає — це суто ботова логіка.

## Env-змінні

Повний список і коментарі — `.env.example`. Ключове:
- `BINOTEL_API_KEY/SECRET`, `BINOTEL_BASE_URL` — ендпоінти підтверджені емпірично на реальному акаунті 2026-07-06 (`stats/list-of-calls-for-period.json` з `startTime`/`stopTime` в unix-секундах, `stats/call-record.json`)
- `OPENAI_ANALYZE_MODEL` — `classifyCall`, `identifyManager`, звіти й відповіді бази знань (дефолт `gpt-4o-mini`). `OPENAI_EMBED_MODEL` — embeddings бази знань (дефолт `text-embedding-3-small`, 1536-dim)
- `SHARED_EXTENSIONS` — comma-separated внутрішні номери спільних слухавок (дефолт `901,902`), для яких менеджер визначається з транскрипту
- `DATABASE_URL` — Neon Postgres
- `TELEGRAM_BOT_TOKEN` — **єдиний бот на все**: `@OBVCarServiceWork_bot` (id 8683695884, новий, чистий, без webhook — перевірено `getWebhookInfo` 2026-07-10). Обслуговує і бота звітності (`getUpdates`), і вихідні алерти інжесту (`sendMessage`) — це не конфліктує. `@obvcarservicebot` (id 8979148940) більше **НЕ** використовується (його webhook належить CarBook — `api.carbook.pro/api/telegram/webhook/7971`)
- `TELEGRAM_CHAT_ID` — owner chat: отримувач алертів + дефолт для звітів і allowlist. `TELEGRAM_ADMIN_CHAT_ID` — опційний окремий чат для алертів. `BOT_ALLOWED_CHAT_IDS` (owner-only, дефолт → `TELEGRAM_CHAT_ID`), `BOT_REPORT_CHAT_ID` (дефолт → `TELEGRAM_CHAT_ID`), `BOT_REPORT_TIMES` (`13:00,19:30`)
- ⚠️ **Бот може писати юзеру лише після того, як той першим натиснув Start** — власник має один раз відкрити `@OBVCarServiceWork_bot` і написати йому, інакше авто-звіти/алерти не дійдуть (403 від Telegram)
