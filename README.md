# Binotel Call Analyzer

Один Render Cron Job (`JOB_TYPE=poll`), що запускається кожні ~15 хв і робить дві речі:

1. Бере нові дзвінки з Binotel з моменту останнього успішного запуску, транскрибує через OpenAI, зберігає в Postgres
2. У кожну годину зі списку `REPORT_HOURS` (за замовчуванням `12,19`, київський час) — генерує звіт по промпту за період **з моменту попереднього звіту** (не за фіксовану добу), збирає всіх менеджерів в **один PDF** (кожен — з нової сторінки, підпис з періодом внизу кожної сторінки), надсилає його в Telegram одним файлом (`sendDocument`, з коротким підписом-caption) і дописує агреговану статистику по менеджерах в Google Sheets. Захищено від повторного надсилання того самого слоту (`app_state.last_report_slot`)

`JOB_TYPE=report` лишається лише для локального ручного форс-тесту (`npm run report`) — на Render не деплоїться.

Менеджер визначається автоматично з `employeeData.name` у відповіді Binotel (підтверджено на реальних даних — напр. "Роман" для 903). Якщо для якогось номера (напр. спільна слухавка 901/902) ім'я не повернулось — падаємо назад на сам внутрішній номер як ідентифікатор. Жодних хардкод-мап імен у коді немає.

Все логується в консоль (`[poll]`, `[binotel]`, `[transcribe]`, `[analyze]`, `[telegram]`, `[processCalls]`). Поки немає `TELEGRAM_BOT_TOKEN`, `telegram.js` не падає, а виводить готовий текст повідомлення в консоль ("DRY RUN").

## Сховище — Postgres (Neon)

1. Зареєструватись на [neon.tech](https://neon.tech) (без картки, 2 хв)
2. Створити проєкт, скопіювати `DATABASE_URL` з дашборду
3. Вставити в `.env` — таблиці (`calls`, `pending_calls`, `app_state`) створюються автоматично при першому запуску (`migrate()`)

Безкоштовного тарифу (0.5 ГБ, 100 CU-год/міс, засинає при простої) вистачить з великим запасом для наших обсягів.

## Надійність — як ми уникаємо провалів у даних

- **Чекпоінт замість фіксованого вікна**: кожен успішний `poll`-запуск запам'ятовує, до якого моменту часу дзвінки вже оброблено (`app_state.last_polled_until`). Наступний запуск завжди продовжує рівно звідти, а не бере "останні 20 хв" — тобто пропущений/затриманий cron-запуск не створює діру в даних. Перший запуск (коли чекпоінта ще нема) бере `POLL_WINDOW_MINUTES` як старт.
- **Черга `pending_calls`**: якщо запис дзвінка ще не готовий (`recordingStatus != uploaded`) або сталась помилка на будь-якому кроці (Binotel/OpenAI недоступні) — дзвінок не губиться, а йде в чергу `pending_calls` і повторно пробується на **кожному наступному** `poll`-запуску, поки не вдасться (або поки не вичерпає `MAX_PENDING_ATTEMPTS`, за замовчуванням 20 спроб — після цього статус стає `failed`, а в Telegram-адмінчат летить сповіщення для ручної перевірки; сирі дані лишаються в таблиці, нічого не видаляється мовчки).
- **Ретраї на кожному зовнішньому виклику**: Binotel, OpenAI (транскрибація й аналіз) і Telegram обгорнуті в `withRetry` (`src/retry.js`) — 2-3 спроби з експоненційною паузою, перш ніж вважати крок невдалим і йти в чергу `pending`.
- **Markdown fallback у Telegram**: якщо звіт від OpenAI містить пошкоджену Markdown-розмітку (незбалансовані `*`/`_`) і Telegram відхиляє повідомлення — скрипт автоматично повторює надсилання без `parse_mode`, звичайним текстом, аби звіт не загубився.

## Сповіщення про збої

`sendAlert()` (`src/telegram.js`) шле повідомлення в `TELEGRAM_ADMIN_CHAT_ID` (або в `TELEGRAM_CHAT_ID`, якщо окремий адмін-чат не задано) у трьох випадках:
- джоба (`poll`/`report`) впала з необробленою помилкою
- дзвінок у черзі `pending_calls` вичерпав ліміт спроб
- не вдалося згенерувати/надіслати звіт для конкретного менеджера

**Залишок коштів на OpenAI** — свідомо не робив через API: ендпоінт перевірки балансу в OpenAI недокументований і може зникнути будь-коли. Натомість надійніше й простіше налаштувати вбудований контроль прямо в кабінеті OpenAI: **platform.openai.com → Settings → Limits** — виставити місячний бюджет і email-сповіщення при досягненні порогу. Це нативна, задокументована функція, нічого додатково підтримувати не треба.

## Перед першим запуском

1. `DATABASE_URL` від Neon (див. вище)
2. `OPENAI_API_KEY` — вже є в `.env`
3. (не потрібно для першого тесту, тільки для звіту) `TELEGRAM_BOT_TOKEN` і `TELEGRAM_CHAT_ID`

## Швидкий тест одного дзвінка (без БД)

```
npm install
npm run test:call -- <generalCallID>
```

## Обробка за конкретний період (backfill)

```
npm run backfill -- "2026-07-01 00:00:00" "2026-07-03 23:59:59"
```

Дати — в локальному часовому поясі машини. Період автоматично розбивається на шматки ≤23 год (ліміт Binotel). Вже оброблені дзвінки пропускаються — можна безпечно перезапускати.

## Повний локальний запуск

```
npm run poll     # обробити нові дзвінки з моменту чекпоінта, плюс ретрай pending-черги
npm run report   # звіт за останні 24 год, вивід у консоль (DRY RUN) або в Telegram
```

## Деплой на Render (без GitHub)

```
docker build -t <твій-докерхаб-логін>/binotel-call-analyzer .
docker push <твій-докерхаб-логін>/binotel-call-analyzer
```

В Render dashboard:

1. New → Cron Job → Existing Image → вказати образ з Docker Hub
2. Створити **два** окремі Cron Job сервіси з одного образу:
   - `binotel-poller`: schedule `*/15 * * * *`, env `JOB_TYPE=poll`
   - `binotel-daily-report`: schedule `0 8 * * *` (щодня о 8:00 UTC), env `JOB_TYPE=report`
3. В обох сервісах додати решту env-змінних з `.env.example` як secrets (включно з `DATABASE_URL` — БД зовнішня, спільна для обох сервісів)
4. Після зміни коду: `docker build` → `docker push` → в Render натиснути "Deploy latest image" (або дернути deploy hook URL)

Вартість: Cron Job тарифікується посекундно, мінімум $1/міс за сервіс — для двох джобів це ~$1-2/міс сумарно.
